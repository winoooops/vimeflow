Chat View — Component Architecture

How should the Chat center column be decomposed into React components?

A

Flat Components (Recommended)

Minimal component tree — a single ChatView page component containing MessageThread, UserMessage, AgentMessage, CodeBlock, MessageInput, and StatusBadge. Simple, matches the design 1:1, easy to lift into the full app shell later.

Pros

  • Direct mapping from HTML reference to React components
  • Easy to verify against screen.png
  • Fewer abstractions = fewer bugs

Cons

  • Some repetition in message styling
  • Refactoring needed when adding real data
B

Design System First

Build a primitive component library first (Surface, GlassPanel, Badge, Avatar, Typography), then compose the Chat view from those primitives. Matches the design system spec more faithfully.

Pros

  • Reusable across all 5 screens later
  • Enforces design tokens from day 1
  • Easier to maintain consistency

Cons

  • More upfront work before seeing anything
  • Risk of over-abstracting for dead pages
  • Primitives may not survive contact with real implementation
C

Hybrid — Tokens + Flat Components

Set up the Tailwind theme config with all design tokens (colors, fonts, border-radius) from the reference HTML, then build flat Chat components using those tokens. No component abstractions, but the design system vocabulary is baked into Tailwind.

Pros

  • Design tokens defined once, used everywhere
  • Components stay simple and direct
  • Tailwind config carries forward to all future screens

Cons

  • Slightly more setup than pure flat approach
