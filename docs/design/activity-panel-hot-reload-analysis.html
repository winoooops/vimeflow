右侧状态面板不应该“跳”：它应该像热视图一样切换

录屏里的问题不是简单的 CSS 抖动，而是 pane 切换时状态从冷启动重新装载：旧内容消失、空态出现、后端 replay 再把长列表补回来。更优雅的解法是把右栏设计成 缓存快照 + 后台预取 + 轻量同步态 的 live lens。

Supporting document

This document supports the Linear planning issue for stabilizing the Vimeflow right-side agent activity panel during rapid pane switching. It is intentionally committed under docs/design/ because the proposed fix changes the activity panel interaction contract, not just implementation plumbing.

Linear issue: VIM-127 · Epic: stabilize activity panel hot reload across pane switching

现象：为什么看起来“超级乱跳”

从录屏看，右侧 activity panel 在不同 pane 之间切换时，会在短面板和长面板之间瞬间重排。有些 pane 已经积累了很多 tool calls、activity events、files changed；有些 pane 几乎为空。当前 UI 把这些内容直接替换到同一个滚动容器里，于是浏览器必须立刻重新计算高度和滚动位置。

关键判断：问题的核心不是“动画不够好看”，而是“切换时进入了错误的状态模型”。 它把已经访问过的 pane 当作第一次打开，导致用户看到冷启动过程。

当前观感

长内容消失 → 空态出现 → replay 填充 → 列表高度变化 → 右栏跳动。

目标观感

旧快照立即出现 → 角落显示同步 → 新事件轻量补齐 → 右栏位置稳定。

根因：状态订阅是 live 的，但切换体验是 cold 的

Vimeflow 的 active pane 会驱动右栏读取对应的 ptyId。当 active pane 改变时，agent status hook 会重新进入目标 PTY 的检测和事件订阅流程。这个流程技术上正确， 但 UI 上会暴露中间态。

1 · active pane

用户切换 pane，右栏目标从 pty-a 变成 pty-b。

2 · reset

前端状态暂时回到默认值：context、tool calls、activity feed 都变空。

3 · replay

后端 watcher / transcript replay 逐步送回历史事件和当前 metrics。

4 · layout

右栏 DOM 从空变长，scroll height 改变，视觉上就是跳。

新的模型：右栏是 live lens，不是详情页重载

右侧面板应该被看成一个“实时透镜”。切换 pane 时，它不应该卸掉全部内容再从零加载， 而应该先显示目标 pane 的最近快照，然后在后台确认 freshness。

Snapshot

按 ptyId 缓存最后一次完整状态。切回 pane 时立即可画。

Syncing

快照可能不是最新，但可以保留；只显示轻量同步标识。

Live

订阅完成后增量更新，不让用户看到空白中间过程。

方案：热缓存、预取、加载态三层组合

1. 建立 agentStatusStore

把 agent status 从“组件局部状态”提升为“按 PTY 分片的状态缓存”。每个 ptyId 一条记录，包含最新 status、freshness、订阅状态、最后更新时间。 

type AgentStatusRecord = {
  ptyId: string
  status: AgentStatus
  phase: 'cold' | 'snapshot' | 'syncing' | 'live' | 'stale'
  updatedAt: number
  hasWatcher: boolean
}

2. 当前 session 的 shell panes 做轻量预取

用户在一个 split view 里频繁切 pane，所以预取范围不需要无限扩大。 优先预热当前 session 中的所有 shell panes，再保留最近访问过的少量 pane。

  • 当前 active pane：完整 live subscription。
  • 同 session 的 sibling panes：轻量 detect + 最新快照。
  • 最近访问 pane：短 TTL 缓存，例如 30-60 秒。
  • 关闭 pane 或 PTY 退出：立即清理缓存和 watcher。

3. loading 不再等于 empty

如果已有快照，切换时不要显示“无活动”。显示旧快照，并在 header 或 metrics 区显示 syncing。只有完全没有快照的 pane 才显示固定高度 skeleton。

避免

No activity yet 先闪出来，然后马上被几十条事件替换。

采用

保留快照内容，追加一个低调的 syncing 标签。

4. 固定 scroll viewport，内容内部更新

Header、context bucket、token cache 这些关键区域应该有稳定尺寸。 Tool Calls、Activity、Files Changed 放在同一个固定的 scroll viewport 中，pane 切换只重置这个内部滚动容器， 不让整个右栏参与高度动画。

5. 动画只做 crossfade，不做高度魔法

高度动画会和长列表、滚动条、replay 事件互相打架。更稳的方式是： pane key 变化时，内容区做 120-160ms 的 opacity + 4px translate，且尊重 prefers-reduced-motion。

建议架构

WorkspaceView
  ├─ resolve activePtyId
  ├─ resolve visibleSiblingPtyIds
  ├─ useAgentStatusStore(activePtyId)
  ├─ useAgentStatusPrefetch(visibleSiblingPtyIds)
  └─ AgentStatusPanel
       ├─ receives record.status
       ├─ receives record.phase
       └─ renders stable viewport

重点是：WorkspaceView 不需要因为每个 tool-call 都重建整棵树。store 可以把高频事件集中到对应 ptyId 的 record 里，面板只订阅当前需要的分片。

落地顺序

1

先做 agentStatusStore，保持现有后端事件协议不变。

2

让 active pane 使用 store snapshot，消除切换时的空白闪烁。

3

加入 sibling panes 预取，只覆盖当前 session，避免过度 watcher。

4

给面板加 phase：cold、snapshot、syncing、live、stale。

5

最后再优化视觉层：固定 viewport、轻量 crossfade、固定 skeleton 高度。

PR 聚合计划

  1. PR 1 — docs + planning branch. Commit this supporting document and the Prettier opt-in. No runtime behavior.
  2. PR 2 — status store. Extract AgentStatusRecord and per-ptyId cache behind a small hook API. Existing active-pane behavior should remain equivalent.
  3. PR 3 — prefetch and freshness. Add bounded sibling-pane prefetch, TTL, cleanup on pane close/restart, and phase semantics.
  4. PR 4 — panel rendering stability. Add stable viewport, snapshot/syncing UI, scroll reset rules, and reduced-motion-safe crossfade.
  5. PR 5 — verification. Add regression tests and browser/recording verification for long-feed to short-feed pane switches.

取舍与风险

  • 不要预取全局所有 pane。 只预热当前 session 和最近访问项，避免 watcher 数量失控。
  • 快照需要标记 freshness。 旧数据可以显示，但不能伪装成 live。
  • 缓存清理要跟 PTY 生命周期绑定。 pane close、restart、agent process change 都要清理或重置。
  • 高频事件要批处理。 tool-call replay 可能密集到达，应该合并到一帧或一个短 debounce 窗口。

最终效果

切换到已经访问过的 pane：右栏立即显示最后快照，角落显示同步态， 最新事件补齐后安静更新。切换到完全新的 pane：显示固定高度 skeleton， 不撑爆布局。长 feed 和短 feed 之间切换时，滚动容器内部变化， 外层右栏不再跳。

简单说：我们不是让跳动变得“动画更漂亮”，而是让右栏不再进入会跳的状态。
