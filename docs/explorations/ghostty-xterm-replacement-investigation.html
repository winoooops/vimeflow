Vimeflow terminal architecture

Replacing xterm.js with Ghostty

A technical investigation into whether Vimeflow should replace its current browser terminal renderer with Ghostty, what would actually have to change, and where the upside is real.

Date 2026-06-15 Worktree docs/ghostty-terminal-investigation Scope research note, no runtime changes

Recommendation

Do not attempt a direct "swap xterm for Ghostty" inside the current React terminal pane. Ghostty is not a drop-in web component, and Vimeflow currently depends on many xterm.js browser APIs beyond text rendering.

The credible path is a staged R&D track: keep xterm.js as the production pane, introduce a terminal-renderer adapter boundary, and prototype libghostty-vt as a parser/state engine with raw PTY bytes. Revisit a full replacement only after the API, renderer story, packaging, and test harness prove out.

01 Current xterm Contract

Vimeflow's terminal pane is not a thin "bytes in, pixels out" wrapper. It uses xterm.js as a browser-native terminal emulator, parser, input bridge, selection model, fit engine, and rendering surface.

Rust sidecar portable-pty, ring buffer, byte offsets, IPC events, session cache, burner foreground polling.
->
Terminal service spawn_pty, write_pty, resize_pty, pty-data, pty-exit, pty-error.
->
xterm.js pane Parser, buffer, WebGL/Canvas/DOM renderer, selection, keyboard input, OSC 7 cwd, fit-to-grid.
Vimeflow surface Current xterm dependency Replacement consequence
TerminalPane/Body.tsx Creates new Terminal(), loads FitAddon, tries WebGL, falls back to Canvas, then DOM rendering. A new engine must expose an equivalent lifecycle, dispose path, renderer fallback story, and host element contract.
Resize and layout FitAddon.fit() calculates columns and rows from a hidden-or-visible browser container. onResize forwards dimensions to the PTY. Ghostty integration must own cell measurement, font loading, hidden-pane behavior, and PTY resize dedupe.
cwd tracking terminal.parser.registerOscHandler(7) parses OSC 7 from shell prompts and agent output. Need an effect/callback path for OSC, plus stale-restore suppression and the existing text-hint fallback.
Restore and replay Terminal.write(data, callback) sequences historical replay, buffered events, and live writes through cursor dedupe. A Ghostty path should consume raw bytes, maintain equivalent replay cursors, and provide a "processed" callback or event.
Clipboard and selection hasSelection(), getSelection(), paste(), selectAll(), and attachCustomKeyEventHandler(). Selection, copy-on-selection, context menu, and paste safety must be rebuilt or bridged through libghostty selection APIs.
Burner terminals Hidden xterm Body instances stay mounted so hide does not lose scrollback or detach from live PTY output. Replacement must support cheap hidden instances or introduce a real snapshot API for remounting.
Production build Vite uses Terser because esbuild mangles an xterm 6.0.0 pattern. Removing xterm could delete that workaround, but any native or WASM Ghostty path introduces its own build/packaging risks.
Important low-level mismatch: Vimeflow currently emits PTY output as lossy UTF-8 strings plus byte offsets. xterm.js can accept strings or Uint8Array, and its decoder is stream aware for byte input. libghostty-vt expects VT bytes, so a serious prototype should add a raw-byte IPC path or a stream-aware decode layer before changing the visible pane.

02 What Ghostty Offers

Ghostty should be treated as three related but different things: the standalone app, the emerging embeddable library family, and the future full-renderer/widget story. Only the library path is relevant to Vimeflow's integrated React workspace.

Standalone app

Fast native terminal

Ghostty is a native terminal emulator for macOS and Linux, using Metal on macOS and OpenGL on Linux. Its own GUI has windows, tabs, splits, native platform integration, and a separate terminfo entry.

This is excellent for end users, but it is not a React component that can replace <Body />.

Plausible R&D path

libghostty-vt

The current library exposes terminal state, VT stream processing, scrollback, reflow on resize, render-state snapshots, formatter APIs, key encoding, mouse encoding, paste utilities, and WASM helpers.

This is the most interesting direction for Vimeflow, but its API is explicitly unstable.

Future path

Renderer/widgets

Ghostty's roadmap mentions later libraries for input handling, GPU rendering, GTK widgets, Swift frameworks, and full terminal views.

Those would matter for a native embedded pane, but they are not the low-risk path today.

Ghostty capability Potential Vimeflow value Current caveat
Parser and terminal state More native-grade VT behavior, reflow, scrollback, state queries, and modern protocol coverage. Requires raw bytes and a new adapter boundary. The C API is new and breaking changes are expected.
Render state API Could drive a custom canvas/WebGL/DOM renderer with dirty-region updates rather than relying on xterm's renderer. Vimeflow would own the renderer, selection painting, IME details, accessibility, and performance tuning.
Formatter to plain/VT/HTML Useful for transcript capture, copy/export, debug snapshots, and test fixtures even without replacing the live pane. Not a replacement for interactive rendering.
Key and mouse encoding Could reduce hand-rolled keyboard/mouse protocol code, including Kitty keyboard and mouse modes. Needs full DOM input mapping, IME handling, platform shortcuts, and focus integration.
xterm-ghostty terminfo More accurate feature advertising if Vimeflow truly behaves like Ghostty. Do not set TERM=xterm-ghostty while xterm.js still renders the pane. Current backend honestly advertises xterm-256color.

03 Integration Options

Option Pros Cons Fit for Vimeflow
Keep xterm.js
production baseline 		Minimal risk, existing tests remain valid, browser integration is already solved, current UX surfaces keep working. Keeps known browser renderer limitations and xterm-specific workarounds. Best default while the UI handoff migration is still active.
Use libghostty-vt as a companion
parser/state experiments 		Lets Vimeflow evaluate correctness, raw-byte handling, HTML snapshots, and state extraction without disrupting the live pane. Requires a native/WASM build path and a stable adapter contract. Strongest near-term R&D option.
Replace renderer with libghostty-vt plus custom UI
canvas/WebGL/DOM renderer 		Could remove xterm renderer quirks while keeping the React shell and Vimeflow pane UX. Vimeflow owns rendering, selection, accessibility, IME, scroll, dirty-region painting, and test fixtures. Feasible only as a major platform project.
Embed native Ghostty renderer/widget
AppKit/GTK/Metal/OpenGL 		Maximum performance and closest to Ghostty's native experience. Electron composes web views through View and WebContentsView, not arbitrary AppKit/GTK terminal widgets. Overlay, focus, z-order, accessibility, and packaging become platform-specific. Not a good match for the current Electron/React architecture.
Launch Ghostty externally
separate app/window 		Quickest way to let a user run a Ghostty terminal at a workspace cwd. Loses integrated panes, agent observability, PTY replay, burner lifecycle, command palette focus, and session persistence. Useful as an "open in external terminal" feature, not as a replacement.

04 Implications

High risk

Frontend contract churn

  • Define a TerminalViewAdapter before any swap.
  • Move xterm-specific calls behind adapter methods.
  • Preserve focusTerminal, selection, copy/paste, context menu, hidden-pane fit, and OSC cwd callbacks.
  • Rework tests that currently mock Terminal, FitAddon, WebglAddon, and CanvasAddon.
Medium risk

Backend byte semantics

  • Replace lossy UTF-8 replay with stream-aware raw bytes or a lossless byte transport.
  • Keep raw byte offsets as the source of replay dedupe truth.
  • Decide whether the renderer, sidecar, or both own terminal state.
  • Avoid setting TERM=xterm-ghostty until the behavior actually matches Ghostty.
Medium risk

Packaging and portability

  • Build Zig/C artifacts or WASM alongside the Rust sidecar.
  • Bundle artifacts for macOS and Linux in Electron Builder.
  • Plan for Windows separately: Ghostty GUI Windows support is not the current stable target.
  • Track ABI/API churn because libghostty-vt is not stable.
Upside

Where the payoff could be real

  • Better VT correctness and modern protocol support.
  • Cleaner terminal-state snapshots for restore, export, and testing.
  • Potentially fewer browser renderer workarounds in the long run.
  • Shared state engine for terminal, logs, transcripts, and agent-status parsing.
Architecture warning: Vimeflow's sidecar is already the PTY owner. Replacing xterm with Ghostty should not move PTY lifecycle ownership into a second terminal app process. Keep the sidecar as the process/session authority and evaluate Ghostty as an emulator/rendering component only.

05 Suggested Plan

Stage Goal Exit criteria
0. Baseline Keep xterm production behavior. Add no user-facing Ghostty dependency. Document current xterm contracts and test fixtures. Fix the deferred backend UTF-8 chunk-boundary issue independently.
1. Adapter seam Introduce an internal adapter around xterm without changing behavior. Existing terminal, burner, restore, clipboard, and resize tests pass with xterm behind the adapter.
2. Raw byte path Preserve PTY bytes losslessly to the renderer or to a terminal state engine. Replay offsets remain correct for split multibyte sequences, invalid UTF-8, and large agent output.
3. Headless Ghostty prototype Feed the same PTY stream into libghostty-vt in a hidden/dev-only path. Compare cursor position, scrollback, title/OSC effects, HTML formatter output, resize reflow, and keyboard/mouse encoders.
4. Visual prototype Build a single-pane proof of concept using render-state snapshots and a Vimeflow-owned renderer. Handles nvim, tmux, Claude/Codex output, selection/copy, paste, hidden/reveal, focus restore, and resize without regressions.
5. Decision record Decide whether Ghostty should replace xterm, augment it, or stay external. Quantified performance/correctness data, packaging plan, and a rollback strategy.
