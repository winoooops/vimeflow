02Migrate the existing ~26 open GitHub issues

Linear ships a native in-product importer for GitHub Issues (alongside Jira/Asana/Shortcut/Trello). It retains far more than a CSV and supports bulk-delete if an import goes wrong.[6]

Path A — in-app importer (recommended). Settings → Import/Export → Import issues → GitHub. Authorize, pick the repo, map labels/state, run.

Path B — CLI (@linear/import, repo linear/linear-import) for scripted/repeatable imports.[9]

Caveats to plan for: ① created/modified dates do not carry over — issues get the import timestamp. ② To preserve author / assignee / @mentions / comment authorship, make sure each GitHub user already exists in Linear with the same email and has linked accounts under Settings → Integrations first. ③ For a solo workspace, just accept that everything imports as you.[6]

Because only ~26 issues are involved, a pragmatic alternative is to not bulk-migrate: close the stale/low-value ones on GitHub (you just triaged them), and re-create only the live handful directly in Linear so they start with clean statuses and IDs. The importer is there if you'd rather lift-and-shift everything.

03Go-forward: assigning tasks & managing status

Assigning & status (the thing you want to learn)

An issue has one Assignee and one Status. You drag a card across board columns or press Cmd/Ctrl+Shift shortcuts; status is a real workflow, not just open/closed. Cycles answer "what's this week"; Projects group the epics.

Let PR activity move status for you — automatically

This is the highest-leverage GitHub link: connect a PR to an issue and Linear moves the status on its own.[8]

  • Branch name — copy the Linear-generated branch name (Cmd/Ctrl+Shift+. on the issue) and use it for your worktree branch.
  • Magic words in PR title/body — closing: fix/fixes/close/closes/resolve/resolves/complete/implements ENG-123; non-closing: ref/refs/related to ENG-123.
  • Automation — PR opened → In Progress; merged to default branch → Done. Customizable per team (incl. branch-regex rules) under Settings → Team → Issue statuses & automations.
  • Two-way Issues sync covers Title, Description, Status, Assignee, Labels, Sub-issues, Comments.

Fit with your workflow: your worktree branches already follow feat|fix|refactor/<name>. Swap in Linear's branch name (or just append ENG-123), and your existing stacked-PR + request-pr/approve-pr flow drives Linear status with zero extra effort.

04Agent autonomy — the core goal

"More autonomy" decomposes into three layers you can adopt independently. Start with A, add B for free, build C when you want true delegation.

Path A — wire the Linear MCP server (do this first)

Remote hosted server at https://mcp.linear.app/mcp (Streamable HTTP, OAuth 2.1, 25+ tools).[2][3]

Claude Code:

claude mcp add --transport http linear-server https://mcp.linear.app/mcp
# then run  /mcp  inside a session to authenticate

Codex (CLI, or config file):

codex mcp add linear --url https://mcp.linear.app/mcp
codex mcp login linear

# ~/.codex/config.toml
[features]
experimental_use_rmcp_client = true

[mcp_servers.linear]
url = "https://mcp.linear.app/mcp"

Now a lifeline skill or a plain prompt can end with "…and set the Linear issue to Done" and the agent does it through MCP — no gh, no API key in your shell.

Path C — programmatic status via the GraphQL API (for scripts/hooks)

Linear's API is GraphQL at https://api.linear.app/graphql; auth via a personal API key (header) or OAuth2; official @linear/sdk (TypeScript).[10] Minimal "move status" mutation a hook could call:

curl -s https://api.linear.app/graphql \
  -H "Authorization: $LINEAR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"query":"mutation{ issueUpdate(id:\"ISSUE_ID\", input:{stateId:\"STATE_ID\"}){ success } }"}'

For true delegation, the Agent platform adds an OAuth actor=app mode that mints a dedicated bot user; request scopes read, write, app:assignable, app:mentionable and subscribe to Agent session events webhooks. Five "agent activity" types let the bot post thoughts/responses back into the issue.[4][5]

Reality check on full hands-off delegation: Anthropic's "assign a Linear issue to Claude Code → trigger a cloud agent session" is an open feature request (claude-code#12925, still open 2026-05-29), not a shipped product. Until it lands, Path C means building the actor=app webhook→agent bridge yourself (or using a third-party agent). Paths A + B give you most of the day-to-day autonomy now with near-zero build.[7]

06Setup checklist (copy-ready)

Phased so you get value after each step. YOU = needs your account/login (an agent can't create accounts or hold your credentials).

  1. Create the workspace & first team. YOU Sign up at linear.app, make a team (name Vimeflow, suggested key VIM). Stay on Free.
  2. Customize statuses & labels. Settings → Team → Issue statuses: confirm Backlog/Todo/In Progress/In Review/Done/Canceled. Recreate your labels (bug, enhancement, security, ui-handoff-migration…).
  3. Connect GitHub (unlocks Path B for free). YOU Settings → Integrations → GitHub → install on the winoooops/vimeflow repo (grant read metadata/checks, read+write issues & PRs). Verify the PR automation rules (open→In Progress, merge→Done).
  4. Migrate or seed issues. Either run Settings → Import/Export → Import → GitHub (lift everything), or hand-create the ~handful of live issues. Remember: dates won't carry; link your GitHub email first if you want authorship preserved.
  5. Add the Linear MCP server (Path A) to Claude Code: claude mcp add --transport http linear-server https://mcp.linear.app/mcp → /mcp to auth. YOU approve the OAuth prompt.
  6. Add it to Codex too: codex mcp add linear --url https://mcp.linear.app/mcp → codex mcp login linear.
  7. Try the loop. In a worktree, name the branch with the Linear ID (or copy Linear's branch name), open a PR with Fixes VIM-1, merge → watch the issue auto-move to Done. Then ask the agent: "set VIM-2 to In Review and comment the PR link" → confirm it lands via MCP.
  8. (Optional) Personal API key for scripts/hooks. YOU Settings → Security & access → Personal API keys; store as LINEAR_API_KEY; use the GraphQL mutation in §4 from a git hook or lifeline step.
  9. (Later) Delegation (Path C). Only if you want assign-to-bot autonomy: build an OAuth actor=app integration with Agent-session webhooks, or wait on claude-code#12925. Not needed for daily use.
