Reading Markdown in the Dock

A technical field guide to the open-source markdown reader/editor landscape — and a concrete plan to give Vimeflow's dock editor a polished reading experience for specs, plans, and design docs, without sacrificing the CodeMirror 6 + vim foundation we already depend on.

Date  2026-05-30 Type  Exploration / tech survey Status  Pre-decision Scope  src/features/editor · DockPanel Data as of  2026-05-30 (npm/GitHub)

⚡ TL;DR

A great reading view cannot live inside CodeMirror today — there is no production-ready in-editor live preview. So the right shape is two complementary surfaces: keep CM6 + vim as the editor, and add a separate reading view rendered by react-markdown. The heavyweight WYSIWYG editors (Milkdown, TipTap, Lexical) all replace CodeMirror and lose vim — a bad trade when reading is the priority.

01 The gap, grounded in our code

Before surveying libraries, here is exactly what happens today when you open a .md spec in the dock.

The dock's editing surface is src/features/editor/components/CodeEditor.tsx, a thin wrapper around CodeMirror 6. It derives a language extension purely from the filename (CodeEditor.tsx:60-63) by calling getLanguageExtension() in src/features/editor/services/languageService.ts. That function maps a handful of extensions to CM6 language packages — and everything else falls through:

src/features/editor/services/languageService.ts:15-33 
switch (ext) {
  case 'js':
  case 'jsx':   return javascript({ jsx: true })
  case 'ts':
  case 'tsx':   return javascript({ jsx: true, typescript: true })
  case 'rs':    return rust()
  case 'json':  return json()
  case 'css':   return css()
  case 'html':
  case 'htm':   return html()
  default:     return null   // ← .md lands here: NO language, NO highlight
}

So a Markdown file is rendered as raw, unstyled monospace text — no heading hierarchy, no rendered tables, no highlighted code fences, no links you can follow. Confirmed two ways:

  • package.json declares CM6 language packs only for css, html, javascript, json, and rust — there is no @codemirror/lang-markdown, and no renderer (react-markdown, markdown-it, remark, …) anywhere in the dependency tree.
  • The FileLanguage union in src/features/editor/types/index.ts names 'markdown', but nothing ever produces a markdown view — it's a label without an implementation.

The editor surface is mounted at DockPanel.tsx:313 (src/features/workspace/components/DockPanel.tsx), which renders <CodeEditor> whenever a file is selected. (Note: .../panels/EditorPanel.tsx is only the 260px sidebar placeholder — "No file open" — not the live surface.) This is our single integration point.

The constraint that shapes everything The editor is not just CodeMirror — it is CodeMirror plus a deeply-wired vim mode (useCodeMirror.ts:164-189): the vim() extension, a global :w→per-view WeakMap save router (useCodeMirror.ts:62-92), and a vim scroll-follow transactionExtender. Any option that replaces CodeMirror for .md files throws all of this away for Markdown. Preserving vim is the lens through which every candidate below is judged.

02 Mental model: four families, one pipeline

"Markdown library" is an overloaded term. The candidates split into four families with very different shapes. Knowing which family a tool belongs to answers most questions before you read a single benchmark.

The four families

  • String renderers (markdown-it, marked) — markdown text → an HTML string. Fast and framework-agnostic, but in React you must inject via dangerouslySetInnerHTML, losing the component tree.
  • React-native renderers (react-markdown, @uiw/react-markdown-preview, streamdown) — markdown text → a React element tree. No innerHTML; every tag is a real React node you can restyle or swap. This is the family built for a reading view.
  • CodeMirror-native (@codemirror/lang-markdown) — adds Markdown highlighting + editing affordances to the editor we already run. Renders nothing, but coexists perfectly with vim.
  • WYSIWYG editors (Milkdown, TipTap, Lexical, ProseMirror) — replace the text editor with a rendered editing surface. Powerful, heavy, and (critically) none ships a viable vim mode.

The framework worth learning: unified (remark → rehype)

The react-native renderers nearly all sit on the unified ecosystem — a "content as a syntax tree" pipeline that is the de-facto standard for Markdown in JS. It is worth understanding because it is what makes a reading view extensible and safe:

raw .md text → remark-parse → mdast → remark-rehype → hast → React elements

mdast = Markdown syntax tree · hast = HTML syntax tree. Plugins hook each stage: remark-gfm (tables, task-lists, strikethrough, autolinks) operates on mdast; rehype-sanitize (XSS allow-list), rehype-highlight (code fences), and rehype-slug/autolink-headings (anchored TOC for long specs) operate on hast. react-markdown is a thin React front-end over exactly this pipeline.

The payoff: a reading view is not a fixed renderer but a composable one. Want GFM tables in plans? Add a remark plugin. Want a sanitized boundary? Add a rehype plugin. Want code fences in our Catppuccin palette? Swap the highlighter. No forking, no innerHTML.

04 The decision

Three coherent architectures. Each is framed as an IDEA block — Intent · Danger · Explain · Alternatives — so the trade is explicit. We recommend Option B (which absorbs Option A).

A

Highlight-only — add @codemirror/lang-markdown

The cheapest possible change: one dependency, one line in languageService.ts. Markdown gets syntax highlighting, code-fence highlighting, list/blockquote auto-continue, and smart backspace — all inside the existing editor.

Intent Make raw-source Markdown editing pleasant with near-zero risk.
Danger It renders nothing. Headings, tables, and links stay as raw syntax — this does not satisfy "polished reading UX." Minor CM6 footgun: ensure a single deduped @codemirror/state.
Explain It's just another LanguageSupport in the language Compartment. vim operates on a different layer (commands/keymap) and is fully preserved.
Alternatives Insufficient alone — but it is the editor half of Option B, so we ship it regardless.
B ✓

Dual surface — CM6 editor + react-markdown reading view

Keep CM6 + vim as the editor (with Option A's highlighting), and add a separate reading view for .md rendered by react-markdown + remark-gfm + rehype-highlight + rehype-sanitize. A toggle in the dock toolbar flips between Source and Reading.

Intent A polished, themeable reading experience for specs/plans/docs and an untouched vim-enabled editor — the literal ask.
Danger Adds the unified/rehype tree (~tens of kB gzip) and one new component + a view-mode toggle. Two surfaces means a small amount of state (which mode per file). All low-risk and reversible.
Explain react-markdown is ESM-only and React-19-native, returns a real element tree (no innerHTML), is secure by default, and its components prop maps every tag to our Catppuccin tokens. The reading view is a plain React subtree in the DockPanel — CM6 and vim are completely unaffected because nothing about the editor changes.
Alternatives @uiw/react-markdown-preview is a faster turnkey route (it wraps react-markdown) if we want GitHub styling out of the box and less control; markdown-it if we ever need a non-React render context.
C

WYSIWYG replacement — Milkdown / TipTap / Lexical

Replace CodeMirror for .md with a rendered editing surface where you edit the formatted document in place (Obsidian-style live preview).

Intent Best-in-class editing of rendered Markdown, if editing — not reading — were the priority.
Danger Forfeits vim for Markdown. No WYSIWYG editor ships a viable vim mode — the only ProseMirror vim extensions are abandoned and unpublished, and Lexical's custom core has none at all. Also the heaviest option (Crepe ~439 kB gzip), and reading is only secondary here. TipTap additionally gates collab/AI/export behind a paid tier.
Explain These are full editor frameworks that own the document surface; they coexist with CM6 for other files but supplant it for .md, taking our @replit/codemirror-vim integration with them.
Alternatives Revisit only if live-preview editing of Markdown becomes a first-class product goal. If so, Milkdown is the pick (Markdown is its native document model → best round-trip fidelity).
Recommendation Ship Option B: react-markdown for the reading view, @codemirror/lang-markdown for the editor. It delivers the polished reading UX directly, keeps the CM6 + vim editor we already rely on, and stays fully reversible. Option C is rejected specifically because it trades away vim for editing power we don't currently need.

05 Integration plan

A concrete path through real files. Each step is independently shippable; vim is never at risk because the editor code path is only ever added to, never replaced.

  1. Add Markdown highlighting (Option A — the free win)

    Add @codemirror/lang-markdown and one branch to src/features/editor/services/languageService.ts. No change to the editor or vim wiring.

    languageService.ts 
    import { markdown } from '@codemirror/lang-markdown'
// …inside getLanguageExtension(filename)'s switch:
  case 'md':
  case 'markdown': return markdown({ codeLanguages: languages })

  2. Create the reading-view component

    A new presentational component beside CodeEditor.tsx. It takes the same content string the editor already receives, so it plugs into the existing buffer with zero new data flow. Theme via a markdown-reading-view class scoped to our tokens.

    src/features/editor/components/MarkdownReadingView.tsx 
    import type { ReactElement } from 'react'
import ReactMarkdown from 'react-markdown'
import remarkGfm from 'remark-gfm'
import rehypeHighlight from 'rehype-highlight'
import rehypeSanitize from 'rehype-sanitize'

interface MarkdownReadingViewProps {
  content: string
}

export const MarkdownReadingView = ({
  content,
}: MarkdownReadingViewProps): ReactElement => (
  <div className="markdown-reading-view min-h-0 flex-1 overflow-auto">
    <ReactMarkdown
      remarkPlugins={[remarkGfm]}
      rehypePlugins={[rehypeSanitize, rehypeHighlight]}
    >
      {content}
    </ReactMarkdown>
  </div>
)

  3. Switch view per file type in the DockPanel

    At DockPanel.tsx:313, branch on the file extension and a view-mode flag. Non-Markdown files are unaffected; Markdown files default to reading and fall back to the real <CodeEditor> (with vim) on toggle.

    src/features/workspace/components/DockPanel.tsx (~:313) 
    const isMarkdown = /\.(md|markdown)$/i.test(selectedFilePath ?? '')

{isMarkdown && viewMode === 'reading' ? (
  <MarkdownReadingView content={content} />
) : (
  <CodeEditor
    filePath={selectedFilePath}
    content={content}
    onContentChange={onContentChange}
    onSave={onSave}
  />
)}

  4. Add the Source ⇄ Reading toggle

    • A segmented control in the dock toolbar, shown only when isMarkdown is true.
    • Persist the chosen mode (per file, or a global default) in the same place the dock already tracks panel state — default Reading for .md so docs open pretty.
    • Optional polish: a vim-friendly keymap entry (e.g. :view / a keybinding) to flip modes without the mouse, keeping the muscle memory intact.

  5. Theme the reading view in the Obsidian Lens

    • Scope a stylesheet to .markdown-reading-view using the tokens in docs/design/tokens.css (--on-surface for body, --primary for headings/links, --surface-container-lowest for code blocks).
    • Wire rehype-highlight to the official Catppuccin Mocha CSS (CSS-variable build) so code fences match the editor's own syntax colors.
    • Add rehype-slug + rehype-autolink-headings later for anchored, TOC-friendly long specs.

  6. Test & verify (project conventions)

    • Co-locate MarkdownReadingView.test.tsx with explicit import { test, expect } from 'vitest' (globals don't satisfy tsc -b / lint-staged).
    • Assert: renders headings/tables/code, sanitizes a <script> payload, and that opening a .ts file still mounts <CodeEditor> with vim.
    • Confirm a single deduped @codemirror/state after adding lang-markdown (npm ls @codemirror/state).

06 Risks, caveats & data provenance

  • Bundle figures are nuanced. bundlephobia counts full dependency trees: @codemirror/lang-markdown shows ~161 kB because it includes view/state/language we already ship — the marginal cost is low-to-mid tens of kB. highlight.js shows ~298 kB for all grammars; the common-language subset is ~37 kB. Always reason about the marginal add.
  • Untrusted input. The docs we render are locally-authored and trusted, but we still add rehype-sanitize by default — cheap insurance if a rendered file ever comes from an agent or a clone.
  • Flagged as unverified: exact gzip sizes for @lexical/react and @lexical/markdown (bundlephobia returned errors at survey time — core only was measured); @uiw/react-md-editor React-19 support is "permissive peer range, no red flags" rather than an explicit declaration. Neither library is on the recommended path.
  • ProseMirror is effectively single-maintainer and its GitHub repos were archived (read-only) on 2026-04-01 with development relocated off-GitHub; packages still publish. This underpins Milkdown and TipTap — not a blocker, but worth knowing if Option C is ever reconsidered.
  • All numbers are point-in-time (2026-05-30) from npm registry + downloads API, the GitHub API, and bundlephobia. Re-check versions before implementing.
