# Agent Status Sidebar ‚Äî Design Spec

**Date:** 2026-04-12
**Status:** Implemented (sub-specs 1‚Äì8 complete, integration WIP)
**Scope:** Real-time agent status panel for the Vimeflow workspace right side

## Overview

Replace the current mock-data-driven `AgentActivity` panel with a real-time agent status sidebar that observes running coding agents (starting with Claude Code). The panel collapses to zero width when no agent is detected and expands to 280px when an agent is active.

## Goals

1. **Auto-detect** when Claude Code is running in the active terminal session
2. **Display real context window status** from the agent itself (not calculated)
3. **Show tool call activity** ‚Äî aggregated summary + collapsible recent history
4. **Show budget metrics** ‚Äî adaptive to API key vs subscription auth
5. **Collapse to zero** when no agent is running, expanding terminal space

## Non-Goals

- Codex, Aider, or generic agent support (future ‚Äî abstraction layer designed for it)
- Controlling or sending commands to the agent
- Full tool call log / transcript viewer (just summary + recent 5-10)

## Architecture: Statusline File Watcher (Approach B)

### Data Sources

| Source               | Data Provided                                                               | Read Method                                                                         |
| -------------------- | --------------------------------------------------------------------------- | ----------------------------------------------------------------------------------- |
| **Statusline JSON**  | model, context window, cost, rate limits, session ID, tokens, lines changed | Rust `notify` crate watches a status file written by a custom statusline command    |
| **Transcript JSONL** | tool calls, messages, file changes                                          | Rust watches `transcript_path` (provided in statusline JSON), tails new JSONL lines |
| **Process tree**     | agent detection (is `claude` binary running?)                               | Rust polls PTY child PID's process tree every ~2s                                   |

### Statusline Bridge

When Vimeflow spawns a terminal session, it configures a custom Claude Code statusline command that:

1. Receives JSON from Claude Code via stdin (standard statusline protocol)
2. Writes it to a predictable path: `<project>/.vimeflow/sessions/<pty-session-id>/status.json`
3. The Rust `AgentWatcher` receives a `notify` file-change event, reads the file, and emits a Tauri event

The statusline command is a shell script that Vimeflow generates per-session:

```bash
#!/usr/bin/env bash
# Generated by Vimeflow for session <id>
cat > /path/to/.vimeflow/sessions/<id>/status.json
```

Claude Code's statusline is configured by setting `CLAUDE_CONFIG_DIR` or injecting `--settings` when launching the agent, pointing to a settings file that includes:

```json
{
  "statusLine": {
    "type": "command",
    "command": "<project>/.vimeflow/sessions/<id>/statusline.sh",
    "refreshInterval": 5
  }
}
```

### Statusline JSON Fields Used

From Claude Code's statusline protocol (all fields optional, safe-access required):

```typescript
interface ClaudeCodeStatusline {
  session_id: string
  model: { id: string; display_name: string }
  version: string
  workspace: {
    current_dir: string
    project_dir: string
    git_worktree?: string
  }
  context_window: {
    used_percentage: number | null // null before first API call
    remaining_percentage: number | null
    context_window_size: number // 200000 or 1000000
    total_input_tokens: number
    total_output_tokens: number
    current_usage: {
      input_tokens: number
      output_tokens: number
      cache_creation_input_tokens: number
      cache_read_input_tokens: number
    } | null // null before first API call
  }
  cost: {
    total_cost_usd: number
    total_duration_ms: number
    total_api_duration_ms: number
    total_lines_added: number
    total_lines_removed: number
  }
  rate_limits?: {
    // only for Claude.ai subscribers
    five_hour: { used_percentage: number; resets_at: number }
    seven_day?: { used_percentage: number; resets_at: number }
  }
  transcript_path: string
  agent?: { name: string }
  exceeds_200k_tokens: boolean
}
```

### Transcript JSONL Parsing

The transcript file at `transcript_path` contains one JSON object per line. Relevant event types to extract:

- **Tool calls**: tool name, arguments summary, result status, duration
- **File changes**: files created/modified/deleted (derived from Write/Edit tool calls)
- **Test results**: parsed from Bash tool calls that run test commands

The Rust `TranscriptParser` tails the file (seeking to end on start, then reading new lines) and emits structured Tauri events.

### Tauri Events

| Event                | Payload                                        | Source                          | Frequency                                          |
| -------------------- | ---------------------------------------------- | ------------------------------- | -------------------------------------------------- |
| `agent-detected`     | `{ agentType, model, sessionId, version }`     | Process tree + first statusline | Once per session                                   |
| `agent-status`       | `{ contextWindow, cost, rateLimits }`          | Statusline file                 | Debounced ~300ms (Claude Code's rate) + 5s refresh |
| `agent-tool-call`    | `{ tool, args, status, timestamp, duration? }` | Transcript                      | Per tool call                                      |
| `agent-disconnected` | `{ sessionId }`                                | Process tree poll               | Once on exit                                       |

## UI Design

### Panel Behavior

- **No agent running**: panel collapses to **0px width** with CSS transition (200ms ease-out). Terminal zone expands.
- **Agent detected**: panel animates open to **280px** (200ms ease-in). Terminal zone shrinks.
- **Agent disconnected**: panel stays visible for 5s showing final state, then collapses.

### Panel Sections (top to bottom)

#### 1. Status Card

- Agent logo/icon (gradient placeholder for now)
- Agent name: "Claude Code" (from `model.display_name` or agent type mapping)
- Status indicator: green dot + "RUNNING" / "PAUSED" / "COMPLETED" / "ERRORED"
- Model badge: e.g., "opus-4-6" (from `model.id`)

#### 2. Budget Metrics (adaptive)

Detection priority: check `rate_limits` first (present = subscriber), then assume API key user.

**Claude.ai subscribers** (when `rate_limits` is present in statusline):

- 5h rate limit bar with percentage (from `rate_limits.five_hour.used_percentage`)
- 7d rate limit bar with percentage (if `rate_limits.seven_day` exists)
- 1x2 grid: API Time, Tokens In/Out combined
- Cost cell hidden (subscription ‚Äî no per-call cost)

**API key users** (when `rate_limits` is absent):

- 2x2 grid: Cost ($), API Time, Tokens In, Tokens Out
- Cost shows `$0.00` until first API call, then updates from `cost.total_cost_usd`
- All from `cost.*` and `context_window.total_*_tokens`

**Fallback** (before first statusline update arrives):

- 1x2 grid: Tokens In (0), Tokens Out (0) ‚Äî placeholder until real data arrives

#### 3. Context Bucket

- Vertical fill gauge (bucket) with gradient fill from `context_window.used_percentage`
- Grid dot pattern background (cosmetic)
- Scale markings on right side (0k ‚Üí max, derived from `context_window.context_window_size`)
- Emoji indicator matching Claude Code's UX: üòä (<60%), üòê (60-79%), üòü (80-89%), ü•µ (90%+)
- Horizontal progress bar below with current token count and max
- Color shifts: primary purple at normal, warning at 80%+, error at 90%+

#### 4. Tool Call Summary (always visible)

- Header: "TOOL CALLS" + total count
- Aggregated chips: one per tool type, showing name + count (e.g., "Read 18")
- Active tool indicator: green left border, pulsing dot, tool name + args path (truncated)
- When no tool is running: no active indicator shown

#### 5. Recent Tool Calls (collapsed by default)

- Header: "RECENT (N)" with expand/collapse chevron
- When expanded: list of last 5-10 completed tool calls
- Each entry: tool name, truncated args, status icon (checkmark/x), duration

#### 6. Files Changed (collapsible)

- File paths in mono font
- Color coding: green (+) for new, blue (~) for modified, red (-) for deleted
- Badge: NEW / EDIT / DEL

#### 7. Tests (collapsible)

- Pass/fail count: "4/5 PASSED"
- Segmented color bar: green segments for passes, red for failures

#### 8. Footer

- Compact single row: session time, turn count, lines added/removed
- Muted styling (smallest text, lowest contrast)

## Agent Detection Flow

```
PTY spawned
  ‚Üí Poll child PID process tree every 2s
  ‚Üí If `claude` binary found:
      1. Set agentType = 'claude-code'
      2. Start watching .vimeflow/sessions/<id>/status.json
      3. On first statusline JSON received:
         - Extract transcript_path
         - Start tailing transcript JSONL
         - Emit agent-detected event with model info
      4. Continue watching both files for updates
  ‚Üí If process exits:
      1. Emit agent-disconnected
      2. Stop file watchers
      3. Frontend holds final state for 5s, then collapses panel
```

## Error Handling

- **Statusline file not found**: agent detected via process tree but no status data yet. Show panel with agent name + "Connecting..." status. Retry watching.
- **Statusline JSON parse error**: log warning, skip update, wait for next file change.
- **Transcript file not found**: show status card + budget metrics without tool call data. Retry when statusline provides `transcript_path`.
- **Agent exits unexpectedly**: show "ERRORED" status with red indicator, hold for 5s, collapse.
- **Multiple agents in same session**: only track the first detected agent. If process tree shows both `claude` and `codex`, prefer the one matching `Session.agentType`.

## Future Extensibility

The architecture supports adding new agents by:

1. Adding a new detector in `AgentDetector` (process name matching)
2. Adding a parser for the agent's status format (if different from statusline)
3. The frontend `AgentStatusPanel` already accepts generic `AgentStatus` types ‚Äî only the data source differs per agent

Codex support would follow the same file-watcher pattern if Codex exposes a status protocol. Aider could use PTY output parsing as fallback.
