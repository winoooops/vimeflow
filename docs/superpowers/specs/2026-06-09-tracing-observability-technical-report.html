Technical Report · June 9, 2026

Correlated Local Tracing for Vimeflow

A purpose-built observability layer now ties a user action in the React renderer to IPC traffic, Rust backend work, and later Claude Code or Codex agent events, with each step persisted as privacy-safe newline-delimited JSON on the local machine.

Branch: feat/tracing-observability Worktree: worktrees/tracing-observability Store: app-data/logs/trace.jsonl Status: committed, verified, reviewed

Executive Summary

I added an opt-in tracing system to Vimeflow that records selected user interactions, correlated backend IPC work, and sanitized agent state transitions without changing the existing agent-status event schema. The renderer still cannot write files; it sends trace-safe metadata to the Rust sidecar, and the sidecar owns durable JSONL persistence in the same app-data area already used by session state.

The first end-to-end path instruments pane rename because it naturally crosses all required boundaries: frontend action, IPC request, backend PTY write, agent adapter event, and result. The design is additive, so broader interactions can be added without rewriting existing diagnostic logging.

Architecture

The implementation centers on a new Rust TraceService and a wrapped TracingEventSink. BackendState owns the service, routes explicit trace IPC commands into it, and uses it to stamp correlated records around backend work. The event sink forwards existing agent-* events unchanged, then records a separate allowlisted trace entry only when tracing is enabled.

Correlation Flow

  1. Frontend startUserInteractionTrace generates correlationId and spanId, then records a frontend/user.interaction event for pane.rename.
  2. IPC request renameAgentSession attaches the same correlation ID and parent span to the rename_agent_session request.
  3. Backend work The Rust runtime records ipc.request, backend.work, and ipc.result with the same correlation context.
  4. Agent event A short-lived session context lets a later agent-session-title event inherit the original user action correlation ID.
  5. Trace file The records are appended to trace.jsonl, making the cross-boundary path queryable with tools such as jq.

JSONL Event Shape

Each trace line is one JSON object with stable top-level fields and an allowlisted attributes map. Prompt text, response text, terminal data, file contents, and tool arguments are not persisted by default. 

{
  "schemaVersion": 1,
  "timestamp": "2026-06-09T00:00:00.000Z",
  "correlationId": "vf_corr_...",
  "spanId": "vf_span_...",
  "parentSpanId": "vf_span_...",
  "layer": "frontend|ipc|backend|agent",
  "event": "user.interaction|ipc.request|backend.work|ipc.result|agent.event",
  "sessionId": "pty-trace",
  "agentType": "codex",
  "status": "ok",
  "attributes": {
    "operation": "rename_agent_session.write_pty"
  }
}

Privacy, Durability, And Retention

  • Tracing is disabled by default and must be enabled through an explicit IPC-backed setting.
  • Data is local-only. No trace data is transmitted off-device.
  • The renderer never writes log files. The Rust sidecar owns all disk writes.
  • Logs live at <app-data-dir>/logs/trace.jsonl, matching the app-data path plumbed by --app-data-dir.
  • Linux default path: ~/.local/share/vimeflow/logs/trace.jsonl.
  • macOS default path: ~/Library/Application Support/vimeflow/logs/trace.jsonl.
  • Windows is intentionally unsupported for this feature.
  • Unix files are created with 0600 permissions and O_NOFOLLOW, extending the existing debug log security posture into release-capable tracing.
  • Rotation is size bounded. The active file moves through .1, .2, and .3, with the oldest generation removed.
  • Newlines and carriage returns are escaped, values are capped, and secret-looking keys or token patterns are redacted.

Next Expansion Points

The vertical slice is deliberately narrow. The next useful expansions would be to add more renderer interactions, provide a small settings surface for enabling tracing, and add a lightweight trace viewer or export command. The core guardrails should remain unchanged: opt-in, local-only, allowlisted attributes, no prompt or response bodies, and no file contents persisted by default.
