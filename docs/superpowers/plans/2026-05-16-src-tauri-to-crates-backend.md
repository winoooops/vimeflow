# `src-tauri/` → `crates/backend/` Rename Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Rename the Rust sidecar source directory from `src-tauri/` to `crates/backend/` and introduce a Cargo workspace manifest at the **repo root** (`./Cargo.toml`). The Rust binary name (`vimeflow-backend`), library name (`vimeflow_lib`), and Cargo package name (`vimeflow`) all stay unchanged — only the directory path moves and the workspace shape gains a root manifest. Closes the final deferred follow-up from the May 2026 Electron migration.

**Architecture:** Mechanical refactor — `git mv` preserves history; a single 3-line root `Cargo.toml` registers `crates/backend` as the workspace's sole member. Cargo's target dir lands at `./target/` (workspace root). `.cargo/config.toml` moves to the workspace root so cargo's discovery walk finds it regardless of which directory invokes cargo. Every path-aware consumer (npm scripts, `electron-builder.yml`, `electron/main.ts`, both CI workflows, `.gitignore`, `.prettierignore`, lint configs, internal doc-comments, project docs) is updated to match.

**Tech Stack:** Rust 1.77.2 (Cargo workspace, `resolver = "2"`). No new runtime dependencies. The post-PR-D3 crate's `[dependencies]` set is unchanged.

**Spec:** `docs/superpowers/specs/2026-05-16-src-tauri-to-crates-backend-design.md`

**Migration roadmap:** `docs/superpowers/plans/2026-05-13-electron-rust-backend-migration.md` — the May 2026 Electron migration. This rename closes the final deferred follow-up (PR-D3 spec §4.5; `electron-migration` phase in `docs/roadmap/progress.yaml`).

**Sequencing:** This branch was created from `dev` at `c5433da` (PR-D3's squash-merge), _before_ PR #212 (docs wrap-up) lands. Pre-flight in Task 0 instructs a rebase onto post-#212 `dev` before any other work.

## Task 2: Adjust `./.cargo/config.toml` `TS_RS_EXPORT_DIR`

**Files:**

- Modify: `./.cargo/config.toml`

The config file moved from `src-tauri/.cargo/` (depth 1: `.cargo/`) to the repo root's `./.cargo/` (depth 0 relative to project). `TS_RS_EXPORT_DIR` with `relative = true` resolves relative to the config file's directory.

Pre-rename:

```toml
TS_RS_EXPORT_DIR = { value = "../src/bindings/", relative = true }
# Resolved: src-tauri/.cargo/../src/bindings/  →  src-tauri/src/bindings/ ✗ (or src/bindings/ if .cargo/ is at repo root?)
```

Wait — the original config at `src-tauri/.cargo/config.toml` with `value = "../src/bindings/"` and `relative = true` resolved to `src-tauri/.cargo/../src/bindings/`, which is `src-tauri/src/bindings/`. That doesn't match what we know works — `src/bindings/` at the repo root is where ts-rs lands files. So there must be a different relative-resolution rule.

The reality: cargo resolves `relative = true` values relative to where Cargo discovers the config. The original Cargo discovered `src-tauri/.cargo/config.toml` when cwd was `src-tauri/` (via `cd src-tauri &&`). The relative path `../src/bindings/` from cwd `src-tauri/` resolves to `<repo>/src/bindings/`.

Post-rename, the config is at `./.cargo/config.toml`. If cwd is the repo root, `value = "src/bindings/"` (no `../`) resolves to `<repo>/src/bindings/` directly.

- [ ] **Step 1: Read the moved config.**

```bash

```

Expected: 1 `[env]` section with `TS_RS_EXPORT_DIR = { value = "../src/bindings/", relative = true }`.

```toml

```

Replace with:

```toml

```
