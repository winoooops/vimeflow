macOS baked-in Ghostty runtime 技术笔记

当前结论：Linux 继续使用 xterm fallback；macOS 进入 Ghostty native surface spike。第一轮不继续修 WTerm DOM，也不复制 Supacode 的完整 Ghostty submodule 构建链，而是先验证一个最小闭环：Vimeflow backend 继续拥有 PTY，macOS native Ghostty surface 只负责输入编码、VT/grid、Metal 渲染、selection 和 resize。

1. 为什么不是继续修 WTerm/libghostty-wasm

WTerm/libghostty-wasm 已经证明可以真实加载 libghostty-vt.wasm， 但它的 renderer 仍是 DOM。问题不是 VT parser，而是字体 shaping、glyph placement、selection、cursor、颜色状态和高频 repaint 都落在 WTerm DOM renderer 上。这个方向继续修会变成“在 DOM 里重写一个 Ghostty renderer”。

Ghostty 官方文档把 native 路线说得很清楚：libghostty 提供核心 terminal emulation、font handling 和 rendering；macOS app 使用 Swift/AppKit/SwiftUI 链接 C API，Linux 使用 Zig/GTK。也就是说，native 体验来自 libghostty runtime + platform surface，不是只来自 libghostty-vt。

懒结论：保留 WTerm/libghostty-wasm 作为 proof/fallback，不再把它作为主渲染质量路线。

2. 最小可行 macOS 入口：libghostty-spm

调研发现 libghostty-spm 已经把 GhosttyKit/GhosttyTerminal 包成 Swift Package，并提供 prebuilt GhosttyKit.xcframework。它保留了 Metal renderer、CoreText 字体、 输入处理、selection、clipboard 和配置系统，同时提供 host-managed I/O backend。 

TerminalView(frame: .zero)
TerminalController(...)
TerminalSurfaceOptions(backend: .inMemory(session))
InMemoryTerminalSession(
  write: { data in /* terminal input -> host PTY */ },
  resize: { viewport in /* cols/rows -> host PTY resize */ }
)
session.receive(data) // host PTY output -> Ghostty surface

这正好匹配 Vimeflow 现有 Rust sidecar PTY contract： spawn_pty / write_pty / resize_pty / kill_pty + pty-data event。也就是说第一轮不需要改 backend ownership，不需要让 Ghostty 自己 spawn shell。

3. Supacode 给出的上限参考

Supacode 证明 Ghostty native agent terminal 是可行的，但它是 macOS native app，不是 Electron renderer。它通过 Tuist foreign build 生成 GhosttyKit.xcframework， 维护 ThirdParty/ghostty submodule，并在构建时应用 out-of-tree patches。

这条路线接近 native 上限，但 carry cost 高：Zig/Xcode 版本、Ghostty patch 维护、资源/terminfo 打包、Swift/AppKit surface 生命周期都要接管。Vimeflow 的第一轮 spike 不应该直接跳到这里。

5. 两种嵌入方式

方式 优点 风险 建议
Native addon 把 TerminalView 挂到 Electron NSView* 最像真正 pane；Electron 官方提供 getNativeWindowHandle()。 Node-API/SwiftPM/AppKit 生命周期复杂，Electron 版本升级可能破。 当前 Phase C 主线，已进入单 pane parent-host proof。
独立 Swift helper window 覆盖在 Electron 上 最容易启动 SwiftPM + GhosttyTerminal。 z-order、focus、移动窗口、multi-pane、屏幕缩放会很脆。 Phase B/C-a 已验证，只用于证明一条真实 Ghostty native runtime 闭环。

当前结论已经从 helper window 前进到 parent NSView。helper window 证明 Ghostty runtime、PTY byte stream、resize 和 cleanup 这条链路成立； 但它无法解决 z-order 和焦点穿透。真正可交付方向必须是 native addon 把 Ghostty TerminalView 挂到 Electron window 的 NSView*。

6. 推荐 spike 顺序

  1. Phase A：standalone Swift smoke。 已放在 docs/exploration/2026-06-27-ghostty-native-macos-runtime/ghostty-native-macos-smoke。它使用 libghostty-spm + InMemoryTerminalSession，已验证 GhosttyTerminal native surface 和 host-managed I/O 的最小闭环。
  2. Phase B：接 Vimeflow PTY IPC。 已让 Swift host 通过现有 Content-Length framed stdio IPC 启动 vimeflow-backend，调用 spawn_pty / write_pty / resize_pty / kill_pty，并把 pty-data 回灌给 Ghostty surface。不改 Rust PTY ownership，也不改 Electron 产品路径。
  3. Phase C-a：Electron helper window embedding spike。 已新增 macOS-only feature flag： VITE_GHOSTTY_NATIVE_MACOS=1。React terminal pane 上报 bounds/focus，Electron main 启动 Swift helper，Swift helper 用 GhosttyTerminal 渲染真实 PTY bytes，并把 input/cols/rows resize 回传给 main，再由 main 调用现有 sidecar IPC。
  4. Phase C-b：真实 NSView/addon integration。 当前正在做这一步：VITE_GHOSTTY_NATIVE_MACOS_PARENT=1 让 Electron main 通过 getNativeWindowHandle() 取得 macOS NSView*，native addon 加载 Swift dynamic bridge， Swift bridge 把 Ghostty TerminalView 作为子视图挂入 Electron window。Linux 和失败路径继续 xterm。

7. Phase C 实现路径

层 文件 职责
React renderer src/features/terminal/components/TerminalPane/NativeGhosttyBody.tsx 和 src/features/terminal/nativeGhosttyClient.ts 只在 macOS feature flag 开启时替换 xterm body；上报 pane bounds/focus，把现有 terminal service 的 PTY output 作为 raw bytes 转发给 native surface。helper 和 parent-host 两条 spike 共享这个 renderer bridge。
Electron preload/main electron/preload.ts、 electron/ghostty-native-helper.ts、 electron/ghostty-native-parent.ts 暴露 window.vimeflow.ghosttyNative，校验 payload， helper flag 下启动 Swift helper；parent flag 下加载 ghostty_native_parent.node，并把 native input/resize event 转成现有 write_pty / resize_pty sidecar 调用。
Swift helper ElectronHostClient.swift 和 GhosttyNativeMacosSmoke.swift --electron-helper 模式下创建 borderless floating GhosttyTerminal window；stdin 接收 Content-Length framed commands，stdout 发回 input/resize/closed events。
Native parent addon native/ghostty-parent/ghostty_native_parent.cc 和 Sources/GhosttyElectronBridge/GhosttyElectronBridge.swift C++ addon 负责 Node-API、dlopen Swift dylib、 Electron NSView* 解包；Swift bridge 负责 AppKit 主线程建 view、frame 映射、GhosttyTerminal I/O。来自 Ghostty IO 线程的 input/resize 回调必须通过 napi_threadsafe_function 回到 Node 主线程。

helper 数据方向是： backend pty-data -> React terminal service -> Electron main -> Swift helper -> GhosttyTerminal。parent-host 数据方向是： backend pty-data -> React terminal service -> Electron main -> Node addon -> Swift bridge -> GhosttyTerminal。输入方向是： GhosttyTerminal input -> Swift helper/addon -> Electron main -> write_pty。 resize 方向是： GhosttyTerminal cols/rows -> Swift helper/addon -> Electron main -> resize_pty。

9. 当前限制

  • helper window 已经不是交付方向：它只保留为 runtime/PTY 证据。真正方向是 parent NSView addon。
  • parent NSView addon 目前仍是单 pane spike。它已经解决 helper window 的 z-order/focus 穿透大方向，但还需要手动确认输入焦点、 resize 跟随、多 pane 切换、窗口关闭和 orphan cleanup。
  • Swift package 现在仍放在 docs/exploration 下，方便 spike 不污染产品结构。确认 parent-host 路线后，应该迁到类似 native/macos 的专用目录，再接正式 build/package 流程。
  • native callback 不能直接从 Ghostty IO 线程调用 JS。所有 input/resize event 都必须通过 Node-API thread-safe function 回到 Node 主线程。
  • PTY output 现在从 renderer bridge 转发给 main/helper，这是为了最快证明 path；最终可以把 backend event 在 main 里直接 fan-out 给 helper，减少 renderer 中转。
  • 本轮没有做产品级 fallback UI。helper 启动失败时因为 feature flag 仍可关闭回 xterm，但还没有用户可见的自动降级提示。

10. 硬性验收

  • Powerline / Nerd Font glyph 不再破碎。
  • 256 色和 truecolor dense transcript 不污染状态。
  • resize 时只由 Ghostty surface 计算 cols/rows，再转发 PTY resize。
  • coding agent 长 transcript 不冻结；PTY read 不被 renderer IPC 背压。
  • copy/paste、selection、focus、IME、快捷键有明确 ownership。
  • 非 macOS、native host 加载失败、feature flag 关闭时全部回到 xterm。
