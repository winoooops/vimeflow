3 · Codex re-verification3 · Codex 复核agrees = true

An independent codex exec pass over the same pinned source confirmed all three core claims: (1) no library-owned split-pane API; (2) two independent TerminalViews are safe with separate controllers; (3) sharing one controller silences the earlier view via the single-slot wakeup closures. Two refinements surfaced: 对同一锁定版本源码的独立 codex exec 复核确认了全部三条核心结论：（1）库不提供自有的分屏 API；（2）两个独立 TerminalView 在各自 controller 下是安全的；（3）共享 controller 会因单槽位 wakeup 闭包导致先挂载的 view 失效。复核补充了两点修正：

  • Tick pump precision — startDisplayLink() schedules a per-coordinator main-queue tick; MSDisplayLink is imported but never shared. Whether ticking is per-coordinator MSDisplayLink or raw DispatchQueue.main.async does not change the verdict: nothing is shared across views. tick 泵表述精度 —— startDisplayLink() 为各自 coordinator 调度主队列 tick；MSDisplayLink 虽被引入但从不共享。无论 tick 实现是 per-coordinator 的 MSDisplayLink 还是裸 DispatchQueue.main.async，结论不变：view 之间不存在共享。
  • retainedBridges array (TerminalController+Config.swift:64) — one controller can fan out config updates to multiple surfaces. This does not lift the single-slot wakeup constraint (the render trigger is still one closure), so the per-view-controller rule stands. retainedBridges 数组（TerminalController+Config.swift:64）—— 单个 controller 确实能向多个 surface 广播配置更新。但这并不解除单槽位 wakeup 约束（渲染触发仍是一个闭包），因此“每 view 一个 controller”的规则依然成立。

HIGH Callback-context use-after-free. Swift's CallbackBox holds the raw C++ context pointer (GhosttyElectronBridge.swift:75-82) and invokes it on input/resize (:101-119); C++ casts it back in the trampolines (ghostty_native_parent.cc:177-184). Freeing the BurnerContext before the Swift child is destroyed is a UAF, not just a leak. (codex round 2) 回调上下文悬垂指针（UAF）。Swift 的 CallbackBox 持有裸 C++ 上下文指针（GhosttyElectronBridge.swift:75-82）并在 input/resize 时调用（:101-119）；C++ 在跳板函数中将其转回（ghostty_native_parent.cc:177-184）。若在 Swift 子终端销毁前释放 BurnerContext，就是 use-after-free，而不只是泄漏。（codex 第二轮） RemoveBurnerChild destroys the Swift child first, then releases tsfns/context; mirror the primary's atomic released flag. RemoveBurnerChild 先销毁 Swift 子终端，再释放 tsfn/上下文；沿用主 surface 的原子 released 标志。
HIGH Child frames in the wrong coordinate space. setFrame y-flips renderer coordinates via parentHeight for the container (GhosttyElectronBridge.swift:261-275); the primary then fills container.bounds (:277). Child sub-rects must be computed in the container's local bounds — a second parent-height y-flip misplaces the burner. (codex round 2) 子 view 坐标系用错。setFrame 用 parentHeight 对渲染进程坐标做 y 翻转来定位容器（GhosttyElectronBridge.swift:261-275）；主终端随后填满 container.bounds（:277）。子矩形必须在容器的局部 bounds 中计算 —— 再做一次 parentHeight y 翻转会把 burner 放错位置。（codex 第二轮） layoutChildren() computes sub-rects purely from container.bounds — no parentHeight involvement. layoutChildren() 只基于 container.bounds 计算子矩形 —— 不涉及 parentHeight。
MED Split-geometry divergence. Two renderer-driven update paths plus the 120 ms resize quiet window (NATIVE_RESIZE_TRACKING_QUIET_MS) can briefly disagree, leaving a gap or overlap; overlapping sibling NSViews also break hit-testing (top view steals clicks). 切分几何不一致。两条渲染进程驱动的 update 路径叠加 120ms 缩放静默窗口（NATIVE_RESIZE_TRACKING_QUIET_MS）可能短暂不一致，产生缝隙或重叠；同级 NSView 重叠还会破坏点击测试（顶层 view 抢走点击）。 Let Swift own the split: one bounds source (the existing per-pane update), layoutChildren() computes both integer-rounded sub-rects. This also satisfies the handoff's “nested divider must be native/AppKit” requirement. 让 Swift 拥有切分权：单一 bounds 来源（现有的按窗格 update），由 layoutChildren() 计算两个整数取整的子矩形。这同时满足 handoff 中“嵌套分隔线必须是原生/AppKit”的要求。
MED Pending-data race. Main buffers PTY output in pendingData when no surface exists yet (ghostty-native-parent.ts:370-377); the burner needs the same buffering between its PTY spawn and addBurnerChild arrival, or early output is lost. 待发数据竞态。surface 尚未创建时主进程会把 PTY 输出缓存在 pendingData（ghostty-native-parent.ts:370-377）；burner 在其 PTY 启动与 addBurnerChild 到达之间同样需要该缓存，否则早期输出会丢失。 Extend the pending-data tail per childId. Codex round 2: the renderer's registerPending buffer drains only when a mounted Body calls onPaneReady (usePtyBufferDrain.ts:60-78) — with the popup gone, the burner needs a new drain owner (see the output-subscriber risk below). 按 childId 扩展 pending-data 缓存。codex 第二轮：渲染进程的 registerPending 缓冲只有在挂载的 Body 调用 onPaneReady 时才会排空（usePtyBufferDrain.ts:60-78）—— 弹窗移除后，burner 需要新的排空持有者（见下方“输出无订阅者”风险）。
LOW tsfn lifecycle leak. If BurnerContext holds its own threadsafe functions, the primary SurfaceHandle's finalizer must also release them. tsfn 生命周期泄漏。若 BurnerContext 持有独立 threadsafe function，主 SurfaceHandle 的 finalizer 必须一并释放它们。 Drain burner tsfns in both RemoveBurnerChild and ReleaseSurfaceCallbacks/FinalizeSurface. 在 RemoveBurnerChild 与 ReleaseSurfaceCallbacks/FinalizeSurface 两处都清空 burner 的 tsfn。
