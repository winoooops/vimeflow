Built-in browser hosting paths for terminal-first agent workspaces

This proposal compares the practical ways Vimeflow can host a real browser inside the Electron workspace, recommends a path for the spike, and sketches the agent-plugin boundary needed for Claude Code, Codex, and the existing claude-in-chrome MCP server to drive browser panes without taking over system Chrome.

Recommended host Electron WebContentsView
Automation surface pane-scoped CDP proxy
Profile model persistent cookies/cache per workspace session
Sync policy app-scoped profile, no Chrome sync

Decision criteria

must support

User-visible browser pane

Browser content must appear as a pane in the same workspace canvas as shell panes, keep audio, support ordinary navigation, and fit Vimeflow's split-pane mental model.

must support

Agent-grade control

External clients need a stable target they can list, attach to, navigate, inspect, and evaluate with Chrome DevTools Protocol semantics.

must preserve

Local security boundary

Remote web content must have no Node integration, no access to Vimeflow IPC, no system Chrome profile reuse, and no Google account sync path. Durable cookies and cache stay inside Vimeflow-owned partitions.

Implementation constraint. The final spike needs a renderer mount point in the existing SplitView empty-slot flow. The proposal can be produced inside the current documentation allowlist, but the working pane requires additive changes to the current SplitView/TerminalZone host files unless the app is refactored first.

Recommendation

Ship the spike on WebContentsView plus app-scoped CDP exposure.

WebContentsView is the only current Electron-native path that is main-process controlled, non-deprecated, compatible with real Chromium webContents behavior, and suitable for a pane that agents can also inspect. The renderer should own intent and geometry only; Electron main should own browser creation, partitioning, navigation, permissions, target identity, and teardown.

  • Create one WebContentsView per browser pane from Electron main.
  • Use a persistent app partition named persist:vimeflow-browser:<workspaceId>:<sessionId> so cookies, HTTP cache, IndexedDB, localStorage, and service worker state survive app restarts for normal developer workflows such as GitHub sign-in, SaaS dashboards, YouTube tutorials, and music/video sites.
  • Keep that persistence app-scoped rather than system-Chrome-scoped: no imported Chrome profile, no Chrome extensions, no Google account sync, and no defaultSession sharing with the Vimeflow shell.
  • Disable Node integration, keep context isolation and sandboxing, and never attach the Vimeflow preload to remote browser content.
  • Expose automation through a Vimeflow-owned CDP-compatible proxy that lists only registered browser pane targets and forwards allowed commands through Electron's Debugger API. Require an explicit per-pane user grant, an ephemeral capability token, and a strict WebSocket Origin allowlist before attachment. Do not expose the trusted Vimeflow shell webContents to external CDP clients.
  • Keep Electron's raw remote-debugging-port dev-only for diagnostics. It is not the production or plugin security boundary because Chromium's target list is app-wide.
  • Use session.setPermissionRequestHandler to deny permissions by default. Ordinary video/audio playback does not require media; keep camera/microphone capture denied unless a future explicit capture prompt is added. Handle mediaKeySystem separately for EME/Widevine checks.

Why not <webview>?

The webview tag is tempting because it fits in JSX, but Electron documents it as not recommended and not guaranteed to remain available. Vimeflow's browser pane is infrastructure for agents, auth, playback, and long-running workspace state, so the lifecycle should be owned by the trusted main process, not a DOM-created guest element.

Widevine risk. The architecture can request mediaKeySystem permission, but stock Electron availability of Widevine-protected playback is a distribution and licensing question. If the current Electron binary cannot create com.widevine.alpha media keys, the smallest honest path is a user decision: adopt a CDM-capable Electron distribution such as a Castlabs-style build, accept a non-DRM playback spike, or keep Widevine evidence delegated to external Chrome while Vimeflow implements the built-in non-DRM browser pane.

Persistent profile and cache policy

A cache-stripped browser is not useful for the primary Vimeflow use cases. Developers need to stay signed into GitHub, cloud consoles, deployment dashboards, documentation sites, video tutorials, and music services while an agent works in the same project. The browser pane should therefore behave like a durable app-owned browser profile, not like an incognito automation tab.

  • Persist by default: keep cookies, HTTP cache, storage buckets, IndexedDB, localStorage, sessionStorage restoration where Chromium supports it, media licenses where supported, and service worker registrations inside the partition.
  • Scope by workspace/session: default to persist:vimeflow-browser:<workspaceId>:<sessionId> so one project can reuse sign-ins without leaking state into unrelated workspaces. A later product setting can opt into a shared persist:vimeflow-browser:global profile for users who want one browser identity across projects.
  • No Google sync: do not reuse system Chrome user data, do not launch system Chrome, do not import Chrome cookies, and do not enable browser account sync. Signing into websites inside the pane is allowed; syncing the browser profile itself is not.
  • Renderer refresh survival: the pane model should be mirrored into app localStorage and the main-process WebContentsView should survive a renderer Ctrl+Shift+R reload. The refreshed React tree remounts the same pane ids and reconnects to the still-live native browser, preserving in-page login state, media playback context, cookies, and cache where Chromium allows.
  • Popup login support: allow bounded window.open login windows for http:, https:, and OAuth about:blank bootstrap popups. These windows must use the same Vimeflow partition as the parent pane, keep Node/preload disabled, and never use the system Chrome profile.
  • User controls: expose clear actions for Clear site data for this pane, Reset workspace browser profile, and Open temporary private browser pane. The private option is secondary and must not be the default.
  • Agent safety: plugin tools can read and operate the active pane only with explicit browser-tool permissions. Persistent authenticated sessions make permission prompts more important, not less.

Spike architecture

1

Renderer chooses pane kind

Empty SplitView slots show Shell and Browser choices. Shell keeps the existing PTY flow. Browser creates a pane with kind: "browser".

2

Main owns WebContentsView

Renderer sends pane identity and element bounds. Main creates or moves the WebContentsView and loads the current URL.

3

Persistent partition

The view uses a Vimeflow partition, not defaultSession and not system Chrome. Cookie persistence is app scoped and session scoped.

4

CDP attaches externally

A chrome-remote-interface client lists the Vimeflow proxy's /json/list, sees only browser-pane targets, calls Page.navigate, and evaluates page state.

Agent-plugin tool registration sketch

The browser feature should expose tools as data, not plugin runtime code. Agents register desired browser capabilities against a pane target; Vimeflow decides which capabilities are available for the pane and maps them to CDP, Electron webContents, or safe UI operations. 

interface AgentToolRegistration {
  pluginId: string
  tools: readonly AgentToolDefinition[]
}

interface AgentToolDefinition {
  name: string
  target: 'active-browser-pane' | 'browser-pane-by-id'
  description: string
  inputSchema: Record<string, unknown>
  permissions: readonly BrowserToolPermission[]
  transport: 'vimeflow-cdp-proxy'
}

type BrowserToolPermission =
  | 'browser.navigate'
  | 'browser.evaluate'
  | 'browser.screenshot'
  | 'browser.read-dom'
  | 'browser.network.read'

const exampleRegistration: AgentToolRegistration = {
  pluginId: 'devtools-lite',
  tools: [
    {
      name: 'browser_evaluate',
      target: 'active-browser-pane',
      description: 'Evaluate JavaScript in the active Vimeflow browser pane and return the serializable result.',
      inputSchema: {
        type: 'object',
        required: ['expression'],
        properties: {
          expression: { type: 'string', minLength: 1 },
        },
        additionalProperties: false,
      },
      permissions: ['browser.evaluate'],
      transport: 'vimeflow-cdp-proxy',
    },
  ],
}
Design boundary. This proposal intentionally stops at the registration shape. The goal's plugin runtime remains design-only; the spike should expose a browser pane and CDP evidence, not implement a plugin host.

Coexistence with claude-in-chrome MCP

  • Keep the existing claude-in-chrome MCP path available for users who want to drive their normal Chrome or a separately launched Chrome profile.
  • Expose Vimeflow browser panes through a separate local CDP-compatible proxy endpoint so MCP clients can distinguish Vimeflow Browser Pane targets from system Chrome tabs without seeing the trusted Vimeflow shell target.
  • Do not import cookies, extensions, Google profile state, or sync data from system Chrome into Vimeflow partitions.
  • Preserve Vimeflow-owned browser cache and site sessions across app restarts so built-in panes remain useful for authenticated SaaS and tutorial/music workflows.
  • Preserve browser pane metadata and native WebContents records across renderer refreshes, and share the same partition with popup login windows so Google/GitHub/OAuth flows can complete without system Chrome.
  • Allow an MCP configuration to prefer built-in Vimeflow panes when VIMEFLOW_BROWSER_CDP_URL and VIMEFLOW_BROWSER_CDP_TOKEN are present, and fall back to its existing Chrome discovery otherwise.
  • Document that only one CDP client may safely assume exclusive automation control at a time, even though Chromium supports multiple clients; Vimeflow should show attachment status to avoid user surprise.

Dependency justification

The browser pane itself should not require a production npm dependency. Electron already provides WebContentsView, Session, permission handlers, webContents, and the Debugger API needed behind a pane-scoped CDP proxy.

  • No new production dependency is justified for the host implementation.
  • Optional dev dependency: chrome-remote-interface is justified only for the requested evidence client because it is the explicit stand-in for claude-in-chrome MCP and can prove list-targets, navigate, and evaluate behavior against the pane.
  • If added, chrome-remote-interface should remain dev-only and should not become the in-app automation abstraction. Vimeflow can talk to Electron's Debugger API or CDP endpoint without shipping CRI to users.
  • A Castlabs-style Electron build is not a normal npm dependency. It is a distribution decision required only if stock Electron cannot satisfy Widevine playback.

Acceptance evidence and current spike result

Current spike evidence was captured with the proxy environment cleared and npm run electron:dev, because npm run dev starts the renderer server only. Files are stored under docs/exploration/2026-05-28-builtin-browser/media/.

  • Pane mount: browser-pane-video-audio.png shows the Browser picker coexistence state and the mounted native browser surface inside the DockPanel/SplitView layout.
  • CDP proxy: cdp-targets.json lists only the registered browser pane target, with the ephemeral local token redacted. cdp-client-navigate-evaluate.txt records an external chrome-remote-interface client attaching, navigating to https://example.com/?vimeflow-cdp=1, and evaluating page state inside the pane.
  • Non-DRM media/audio: nondrm-video-audio-state.txt records HTML5 video playback with paused=false, muted=false, volume=1, readyState=4, and advancing currentTime. This verifies Electron audio output for the embedded browser path.
  • YouTube: youtube-playback-state.txt records YouTube loading in the fresh app-scoped profile, but the page returns "Sign in to confirm you're not a bot". youtube-embed-state.txt records the embed path failing with "Error 153". The shipped persistent partition is therefore required for real YouTube evidence: a user must sign in once inside the Vimeflow browser profile, then reuse that cache/cookie state.
  • Widevine: widevine-capability-state.txt records stock Electron returning NotSupportedError for com.widevine.alpha. The accepted path is to keep the built-in non-DRM browser pane on stock Electron and require a CDM-capable distribution decision before claiming Widevine playback.
