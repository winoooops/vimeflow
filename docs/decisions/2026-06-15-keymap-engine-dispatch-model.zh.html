VIM-136 · SP1 · 调度架构决策

让快捷键"可改"，到底难在哪？

键位自定义引擎的核心抉择 —— 在动手写 spec 之前，先把利害关系讲清楚。

分支 feature/vim-136 基线 feat/settings @ dbe0367 前序 VIM-104 / #460 决策 方案 A / B / C

VIM-104（#460）只交付了预设切换和一个只读的键位面板，把真正的"自定义引擎"整块推迟了。VIM-136 就是来补这个引擎。它的验收线只有一句话，但分量很重 —— 「改一个绑定 → 实时生效，且重启后依旧生效」。 这句话听起来像"接根线"，实际上要动到整个键盘调度的地基。这篇笔记解释为什么，以及三种走法各自的代价。

01现状：键位是"焊死"的

今天每一个全局快捷键，都是一个独立的 hook，各自在 document 上挂一个 capture 阶段的 keydown 监听器，并且把"匹配哪个键"写死在代码里。比如命令面板的 ⌘;： 

// src/features/command-palette/shortcutConfig.ts
export const isCommandPaletteToggle = (event, modifier) => {
  const expectedModifier =
    modifier === 'meta' ? event.metaKey && !event.ctrlKey
                          : event.ctrlKey && !event.metaKey
  return expectedModifier && !event.altKey && !event.shiftKey
                          && event.key === ';'   // ← 键名写死
}

再比如 ⌘1–4 聚焦面板：

// src/features/terminal/hooks/usePaneShortcuts.ts
const digitMatch = /^Digit([1-4])$/.exec(event.code)  // ← 物理键位写死
// …随后是一大段 guard：dialog 守卫 / 终端容器是否聚焦 /
//   越界面板 / 布局容量 / 已激活则放行 …

而设置面板里那张键位表 （sections.ts 的 KEYMAP_GROUPS）只是给人看的文档，靠人手与 hook 保持同步 —— 它根本不驱动行为。这种"两份真相"已经出过 bug： codex 在 #460 抓到键位表在 Linux/Windows 上把修饰键错显成 ⌘（finding #23），因为展示串和行为侧各写各的。

02要"可改键"，缺的是什么

本质上，我们需要把"匹配哪个键"从写死变成查表：一张 数据驱动的绑定表 —— 默认值 ⊕ 用户覆盖，hook 不再硬编码键名，而是问这张表。 用户改了绑定 → 写进覆盖层 → 持久化到 settings.json → hook 下次读到新值 → 行为实时变。

"读表"本身不难。真正难的，是读表的同时不能碰坏 guard。

03真正的风险：那 29 条 guard 教训

docs/reviews/patterns/keyboard-shortcut-guards.md 里沉淀了 29 条 来自历次 review 的踩坑记录。它们说明：每个 hook 的 guard 都是逐条用 bug 换来的、平台相关的、彼此不同的精细逻辑。随手列几条最致命的：

  • #27 · 必须用 event.code（物理键位）而非 event.key（逻辑字符），否则在西里尔/阿拉伯/CJK 输入法布局下整个快捷键静默失效。
  • #17 · 主进程（Electron before-input-event）和渲染进程的 guard 必须对称。主进程 preventDefault() 会吃掉渲染进程的 keydown，所以只在一侧加 guard 等于没加。
  • #26 · 方向键导航必须带 Shift，否则在 Linux/Windows 抢走裸 Ctrl+方向键（shell/vim/readline 的词移动）。
  • #21 · 即使没有命中目标，capture 阶段也要 preventDefault，否则 modified-arrow 转义序列会泄漏给 PTY。
  • #29 · 每个分支必须显式 return，否则后插入的分支会在每次按键时误触。
  • #13 / #20 / #22 · DIALOG_SELECTOR、终端容器聚焦、CodeMirror 聚焦等守卫，差一个就会从编辑器/弹窗里"偷键"。
★ 关键洞察

把任意一个 hook 拆开看，它 = 「键匹配」 + 「guard + 动作」。引擎要替换的只有「键匹配」这一小块；guard 和动作可以原封不动。

这个观察，把"高风险大重构"变成了"外科手术式的替换"—— 也正是下面三种方案分野的根源：你愿意改动多大半径，来换取多统一的调度模型？

04三种走法

方案 A · 共享匹配器，保留每个 hook推荐

每个 hook 的监听器、guard、动作全部不动，只把那一行写死的键匹配换成一次查表 + 一个共享匹配函数： 

// 引擎提供：
const chord = bindingFor('palette')         // 默认 ⊕ 覆盖，解析后的绑定
eventMatchesChord(event, chord, platformMod)  // 纯函数，封装 ⌘-only / event.code / 禁用修饰键

// hook 里原来的：  event.key === ';'
// 变成：
if (eventMatchesChord(event, bindingFor('palette'))) { /* …原有 guard/动作不变… */ }
优点
  • 外科手术式改动，完整保住 29 条 guard
  • 增量迁移：一个 PR 迁一个 hook，未迁的照常工作
  • 回归风险最低
  • 照样 100% 满足验收线（覆盖 → 实时生效 → 持久化）
缺点
  • 仍是 N 个监听器，没有"全局统一调度"
  • 没有全局 event→command 反查表 （但用不上：冲突在保存时拦 + 独立纯函数检测）

方案 B · 中央 keydown 路由重构量大

只留一个 capture 监听器统管所有按键：event → commandId 查表，再调用该 command 注册的 handler。hook 退化为 registerCommand(id, handler, guards)。

优点
  • 单一调度真相，概念最干净
  • 冲突解析集中在一处
  • capture / ⌘-only / dialog 守卫单点强制
缺点
  • 大爆炸式重构：要把异构 guard（终端聚焦 / dock / CodeMirror / 越界面板 / 布局容量）全部泛化进路由
  • 对 29 条 guard 冲击最大，回归风险最高
  • 难增量：路由必须先吃下所有 command，才能拆掉任何一个 hook

方案 C · 混合（路由 + 保留 bespoke）

中央路由只管"守卫统一"的简单全局键（面板 / 新建会话 / 侧栏 / dock / 会话切换 —— 只需 DIALOG_SELECTOR）； 复杂的（面板聚焦/布局、dock）保留各自 hook，但共用同一个匹配器。

优点
  • 把容易的大多数集中掉
  • 难搞的 guard 留在原地，不动其测试
缺点
  • 两套调度路径并存，心智负担
  • "哪些归路由、哪些保留"是会漂移的主观切线

06我的建议：A

不是因为 A "省事"，而是因为在我们这个具体场景里，A 的取舍最对：

  • 最尊重既有资产。 那 29 条 guard 是真金白银的 bug 换来的。B 要把它们连根拔起重新泛化，是拿最稳的部分去冒险。
  • B 的统一性收益，在 guard 高度异构时并不划算。 中央路由的价值在于"所有键走同一条路"；可我们的键恰恰不一样—— 有的看终端聚焦，有的看 dock/CodeMirror，有的看面板越界和布局容量。强行收进一个路由，等于把 if 分支从 hook 搬进路由，复杂度没消失，风险却集中爆发。
  • 冲突检测本就该是独立纯函数（SP2 的编辑 UI 来消费），不需要靠中央路由在运行时兜底。A 用"保存时拦截 + 纯函数检测"就够，而且更好测。
  • 增量、可中断。 A 让 SP1 可以"迁核心几个 hook 即收"，剩下的在后续 PR 跟进，每步都能独立评审、独立回归。
↺ 可逆性

选 A 不堵死 B。如果将来真需要全局统一调度，A 沉淀下来的 命令目录 + chord 表示 + 匹配器 + 解析器 正是 B 的全部地基 —— 届时再把监听器收拢成路由即可。反过来，先上 B 则很难回退。

07A 方案下，引擎由哪些零件组成

SP1 要交付的（每个都小、可单测、可单独迁移）：

  1. 命令目录：可重绑命令的 ID + 默认 chord + 元信息（标签/分区/是否可改/允许的修饰键）。取代 KEYMAP_GROUPS 成为单一真相，展示也从它派生 —— 顺手消灭"两份真相"漂移。
  2. chord 表示：归一化结构 { code, mods } + 'Mod' 抽象（mac=⌘ / 其它=Ctrl），用 event.code 保证布局无关。
  3. 覆盖存储：customKeybindings: Record<CommandId, Chord | null>，落到 Rust AppSettings 新字段（+ ts-rs 绑定重生成）。
  4. 解析器：resolve(defaults, overrides) → Map<CommandId, Chord>，纯函数。
  5. 匹配器：eventMatchesChord(event, chord, platformMod)，把今天散落各 hook 的 ⌘-only / 禁用修饰键 / event.code 逻辑收成一处纯函数。
  6. 冲突检测：detectConflicts(resolved) → Conflict[]，纯函数，供 SP2 用。
  7. 前端接入：useKeybindings() 暴露 resolved + bindingFor(id)，由 useSettings() 喂数据。
  8. 迁移：各 hook 改用 bindingFor(id) + 匹配器；guard/动作不动。

对后续子项目的铺垫：

  • SP2（编辑 UI） = 消费「命令目录」+「冲突检测」，铅笔捕获 chord → 查冲突 → 写覆盖。
  • SP3（可定制 leader） = ⌘; 就是目录里的一个命令；难点不在引擎，而在它要联动三处（渲染进程 / Electron 主进程 / Linux 全局快捷键）。
  • SP4（预设） = VS Code / JetBrains / Custom 不过是命名的覆盖集，叠在引擎之上。
