VIM-136 · SP1 · 最后一个架构抉择

registry 拿到 hook 之后，到底做不做事？

research 已经定了"registry 只存 id+combo、不存 hook"。剩下的唯一问题是：那个 keydown 监听器归谁 —— 组件，还是 registry？

已定 · 引擎方案 A 已定 · 迁移 代表性子集先行 已定 · 持久化 settings.json(非 SQLite) 待定 · 谁拥有监听器

你最初的草图里有一句 register(id, hook, combo)。research 已经确认：hook 绝不能塞进 binding entry（Zed 和 VS Code 都是 id 进表、handler 另放）。 但这句话还藏着一个更深的问题 —— registry 既然能拿到 hook，它拿去干嘛？ 是自己挂监听器、匹配到就回调 hook，还是压根不碰 hook、由组件自己监听、只来查一下绑定？ 这就是下面三个选项的分野，也是 SP1 唯一还没拍的架构点。

★ 一句话区分

三个选项都能满足你的第 1~3 点（一张中心 registry、不懂 hook 细节、提供改键方法 + 默认不动）。 它们唯一的区别是：监听器在谁手里，以及 那 29 条 guard 待在哪。

A选项一 · consult-only（registry 是纯数据）推荐

registry 只有数据 + 改键方法，完全不碰 hook。每个组件 保留自己的监听器和全部 guard，只把"键匹配那一行"从写死换成查表。 

// 引擎：registry 只有数据 + 改键方法，永远拿不到 hook
class KeybindingRegistry {
  // 内部: Map<id, { defaultCombo, combo?, context }>
  bindingFor(id): Chord            // combo ?? defaultCombo
  setUserBinding(id, combo): void  // 写覆盖 → 持久化到 settings.json
  reset(id): void                 // 删覆盖 → 回到默认
  detectConflicts(): Conflict[]     // 纯函数, 供 SP2 编辑 UI 用
}

// 组件里的 hook —— 监听器 / guard / 动作 全部不动, 只换"键匹配"这一行
useEffect(() => {
  const onKeyDown = (e) => {
    // …… 原有的一大堆 guard: dialog / 终端聚焦 / 越界 / 布局容量 ……（不动）
    if (eventMatchesChord(e, registry.bindingFor('palette'))) {  // ← 原来是 e.key === ';'
      e.preventDefault(); openPalette()
    }
  }
  document.addEventListener('keydown', onKeyDown, { capture: true })
  return () => document.removeEventListener('keydown', onKeyDown, { capture: true })
}, [])

在这个模型里，你的 register(id, hook, combo) 自然退化成"声明这个命令 + 它的默认组合"；hook 由组件自己消费，registry 永远拿不到它。 如果你喜欢 register 这种写法，可以包一层组件侧的 useKeybinding(id, handler)（在宿主组件里挂监听器 + 查 bindingFor），但 hook 依旧不进 registry。

优点
  • 改动半径最小：每个 hook 只改一行
  • 29 条 guard 零风险（原封不动）
  • 简单 / 复杂命令同一条路，无第二套 dispatch
  • 依然 100% 满足你第 1~3 点
缺点
  • 仍是 N 个监听器（没有"全局统一调度"的那种概念洁癖）
  • register() 不是字面的"registry 帮你调度"，而是组件侧薄封装

B选项二 · central dispatch（registry 拥有监听器）风险高

registry 自己挂一个监听器，匹配到就回调注册进来的 hook —— 字面实现你的 register(id, hook, combo)。

class KeybindingRegistry {
  register(id, hook, combo) {       // ← 你最初想要的签名, 字面实现
    this.bindings.set(id, { combo, defaultCombo: combo, context })
    this.handlers.set(id, hook)     // hook 存独立 map(不在 binding entry 里)
  }
  // 内部唯一的监听器:
  onKeyDown(e) {
    for (const [id, chord] of this.resolved())
      if (eventMatchesChord(e, chord)) {
        // ❗ 问题: guard 在哪? dialog / 终端聚焦 / 越界 / 布局容量……
        //    必须搬进 hook, 或变成 registry 要"看懂"的 when 谓词
        this.handlers.get(id)(e); break
      }
  }
}
⚠ 风险所在

那 29 条 guard 现在要么逐条塞进每个 hook，要么变成 registry 必须理解的 context 谓词 —— 后者恰恰违背你的第 2 点（registry 不懂 hook 细节）。而且冲击最大的，正是 usePaneShortcuts 这种最复杂的 hook（终端聚焦 / 越界面板 / 布局容量 / CodeMirror）。 这等于拿"engine-first 最想保护的部分"去冒险。

C选项三 · 混合两条路

简单、守卫统一的命令（面板 / 新建会话 / 侧栏 / dock / 会话切换 —— 只有 DIALOG_SELECTOR）走中央 dispatch，给你字面的 register()；复杂的（pane 聚焦/布局、dock）保留选项一的 consult-only。

优点
  • 简单的大多数能用上字面 register()
  • 难搞的 guard 留在原地，不动其测试
缺点
  • 两套 dispatch 路径并存，心智负担
  • "哪些归路由、哪些保留"是会漂移的主观切线

✓我的建议：选项一（consult-only）

  • 它不牺牲任何东西。 你要的"中心 registry + 改用户键的外部方法 + 默认不动"，靠 setUserBinding / reset / bindingFor / detectConflicts 这几个方法全部满足 —— 这就是你的第 3 点。
  • 它最尊重 engine-first 的初衷。 我们选 engine-first 就是为了"不碰 guard"。选项一让每个 hook 只改一行键匹配，guard / 动作 / 监听器生命周期全不动。
  • register(id, hook, combo) 不必丢。 它在选项一里活成组件侧的 useKeybinding(id, handler) 薄封装；和选项二的真正区别只有一个 —— 谁拥有监听器。选项一让组件继续拥有（连带它那套精密 guard），registry 只当"绑定真相"的数据源。
