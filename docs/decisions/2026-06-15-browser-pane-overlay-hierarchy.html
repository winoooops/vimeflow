Vimeflow technical note

Make the Browser Pane Share the Workspace Overlay Hierarchy

The current integration hides native browser content through a workspace-level boolean that manually names every overlay source. It works, but it makes the browser pane a special case. The stronger shape is an overlay registry that browser panes subscribe to like any other pane-level surface.

Short Answer

It is possible for the browser pane to share the same overlay hierarchy as terminal panes, but not by relying on CSS z-index over the native page. Electron WebContentsView is outside the React DOM. React overlays can only beat it if the main process hides, moves, or reorders the native view, or if the overlay itself becomes native.

Recommended path: keep WebContentsView, remove the hand-built areBrowserPanesOccluded list, and introduce a workspace OverlayStackProvider. Overlay primitives register their plane and native-view policy. BrowserPane registers its native content rectangle and derives occlusion from that shared stack.

Current Shape

Today the browser pane already lives inside the split-pane model, but its native page area is controlled through a parent-computed flag: WorkspaceView builds areBrowserPanesOccluded, TerminalZone passes it into SplitView, and SplitView passes it into BrowserPane as isOccluded. 

// src/features/workspace/WorkspaceView.tsx
const areBrowserPanesOccluded =
  terminalFitDeferred ||
  showUnsavedDialog ||
  commandPalette.state.isOpen ||
  hasVisibleBurner ||
  paneRenameNode !== null ||
  fileError !== null ||
  infoMessage !== null

// src/features/browser/components/BrowserPane.tsx
const visible =
  isActiveRef.current &&
  !isOccludedRef.current &&
  rect.width > 0 &&
  rect.height > 0

void setBrowserPaneBounds({ sessionId, paneId, bounds, visible })

The Electron controller then projects visible: false to zero-sized native bounds. That is the correct class of workaround for DOM overlays, but the source of truth is too far away from the overlay system. 

// electron/browser-pane.ts
private applyRecordBounds(record: BrowserPaneRecord): void {
  const active = this.activeTab(record)
  for (const tab of record.tabs.values()) {
    tab.view.setBounds(
      tab === active
        ? visibleBounds(record.lastBounds, record.visible)
        : { x: 0, y: 0, width: 0, height: 0 }
    )
  }
}

The pain is not the visibility mechanism. The pain is that every new dialog, popup, rename surface, banner, drag layer, or future overlay must remember to update a browser-specific boolean. That is a hidden coupling between workspace overlay hierarchy and a single pane type.

Electron Facts

Fact Implication for Vimeflow
WebContentsView is a main-process native View, not a DOM node. React portals and Tailwind z-index cannot directly stack above it. Renderer and main must coordinate bounds or visibility.
View.addChildView(view, index) can reorder native child views, and re-adding an existing view makes it topmost. Native overlays are technically possible, but only if the overlay is also represented as a native view/window.
View.setVisible(false) exists, and bounds can also be set to 0x0. The current zero-bounds strategy can become an implementation detail behind a cleaner native-surface contract.
Electron recommends avoiding <webview> for new work. Moving to a DOM custom element just to make z-index feel simpler trades a known native-view problem for stability and security risks.
Offscreen rendering can provide bitmaps or shared textures. It can make browser pixels part of the DOM/canvas hierarchy, but then input, IME, accessibility, video, and performance become product-level problems.

Target Contract

BrowserPane should not be told by WorkspaceView that overlays exist. Instead, it should participate in the same overlay hierarchy the terminal canvas already lives under.

Overlay sources self-register

Dialogs, popovers, drag covers, command palette, rename controls, and banners declare their overlay plane and whether they occlude native browser surfaces.

Native surfaces subscribe

BrowserPane registers its page rectangle as a native surface and receives occluded: true when any higher overlay plane intersects or globally covers it.

Proposed Type Shape

type OverlayPlane =
  | 'pane-chrome'
  | 'popover'
  | 'dialog'
  | 'palette'
  | 'drag'
  | 'toast'

type NativeOcclusionPolicy = 'none' | 'intersects' | 'global'

interface OverlayDescriptor {
  id: string
  plane: OverlayPlane
  isOpen: boolean
  nativeOcclusion: NativeOcclusionPolicy
  getRect?: () => DOMRectReadOnly | null
}

interface NativeSurfaceDescriptor {
  id: string
  owner: 'browser-pane'
  getRect: () => DOMRectReadOnly | null
  belowPlane: OverlayPlane
}

Occlusion Sequence

CommandPalette opens
  -> useOverlayRegistration({ plane: 'palette', nativeOcclusion: 'global' })
  -> OverlayStackProvider recomputes native surface visibility
  -> BrowserPane receives occluded = true
  -> BrowserPane sends setBrowserPaneBounds({ visible: false })
  -> BrowserPaneController applies zero bounds or setVisible(false)
  -> React palette is now visually and interactively topmost

CommandPalette closes
  -> overlay descriptor is removed
  -> BrowserPane receives occluded = false
  -> BrowserPane sends latest measured bounds with visible = true
  -> controller restores active tab bounds and focus policy decides refocus

BrowserPane Pseudocode

export const BrowserPane = (props: BrowserPaneProps): ReactElement => {
  const contentRef = useRef<HTMLDivElement>(null)
  const nativeSurface = useNativeSurface({
    id: `browser:${props.session.id}:${props.pane.id}`,
    owner: 'browser-pane',
    getRect: () => contentRef.current?.getBoundingClientRect() ?? null,
    belowPlane: 'pane-chrome',
  })

  const syncBounds = useCallback((): void => {
    const rect = contentRef.current?.getBoundingClientRect()
    if (!rect || !nativePaneReadyRef.current) {
      return
    }

    void setBrowserPaneBounds({
      sessionId: props.session.id,
      paneId: props.pane.id,
      bounds: rectToBounds(rect),
      visible:
        props.isActive &&
        props.pane.active &&
        !nativeSurface.occluded &&
        rect.width > 0 &&
        rect.height > 0,
      shortcutContext: shortcutContextRef.current,
    })
  }, [props.isActive, props.pane.active, props.pane.id, nativeSurface.occluded])

  useLayoutEffect(syncBounds)

  return <div ref={contentRef} data-testid="browser-pane-content" />
}

Technical Difficulties

Difficulty Why it matters Mitigation
DOM and native view are different compositors A React overlay can be visually under a native browser page even when its CSS z-index is higher. Treat native browser content as a registered surface with explicit visibility controlled by the main process.
Overlay ownership is distributed Command palette, dialogs, burner terminals, rename affordances, banners, and drag covers live in different components. Each overlay registers itself. No parent manually knows the full list.
Rect measurement is fragile ResizeObserver catches size changes, but pure position changes can require render-time syncing. Keep the existing post-render syncBounds() pattern, add a shared requestAnimationFrame scheduler, and de-dupe bounds keys.
Focus return can steal user intent When a dialog closes, the browser page may reclaim focus while the user expected the terminal zone, dock, or command palette entry path to finish. Keep BrowserPane's existing focus guards and only refocus native content when the pane was active before occlusion and no higher priority focus target requested restoration.
Pointer capture during drag Native web content can consume mouse events while split handles, dock handles, or sidebar handles are dragging. Register drag layers as nativeOcclusion: 'global'. Do not require every drag source to call browser-specific code.
Multiple browser tabs per pane Only the active native tab should receive real bounds. Inactive tab views must stay alive but hidden. Keep BrowserPaneController.applyRecordBounds() as the authority for active-tab projection.
Security boundary must stay strict Remote pages are untrusted. A simpler DOM embed can reopen Node, preload, or navigation risks. Keep sandboxed WebContentsView with nodeIntegration: false, contextIsolation: true, and existing navigation/popup policies.
Testing real layout requires Electron JSDOM cannot prove native views are hidden under real overlays. Add unit tests for registry math, renderer tests for bridge calls, and one Electron visual/integration check for overlay occlusion.

Alternatives Analysis

Option A - Overlay registry with BrowserPane native-surface subscription recommended

Keep the current WebContentsView browser architecture, but replace the parent flag with a generic overlay/native-surface registry.

IDEA

I - Intent
Let BrowserPane share the workspace overlay hierarchy without making every overlay source remember browser-specific gates.
D - Danger
The registry becomes infrastructure; if descriptors are not registered by all overlay primitives, native occlusion can drift.
E - Explain
This preserves the working native browser surface and moves the coupling into one explicit contract: overlays describe their plane, native panes subscribe.
A - Alternatives
Use Option B only if Vimeflow needs native overlays above native pages without hiding the page. Use Option E only as a short compatibility bridge. 
interface OverlayStackApi {
  registerOverlay(descriptor: OverlayDescriptor): () => void
  registerNativeSurface(descriptor: NativeSurfaceDescriptor): NativeSurfaceState
}

const occluded = overlays.some((overlay) =>
  overlay.isOpen &&
  overlay.nativeOcclusion !== 'none' &&
  isHigherPlane(overlay.plane, surface.belowPlane) &&
  (overlay.nativeOcclusion === 'global' ||
    rectsIntersect(overlay.getRect?.(), surface.getRect()))
)

Overlay opens

Option B - Native overlay compositor in Electron main powerful

Move the app shell and overlay surfaces into a native BaseWindow view hierarchy, or add a dedicated overlay WebContentsView above browser page views.

IDEA

I - Intent
Make overlays literally native siblings above browser native views, so the browser can remain visible under translucent glass surfaces.
D - Danger
This is a windowing architecture change. It risks focus bugs, lifecycle leaks, cross-platform child-window behavior, and a large test matrix.
E - Explain
Electron supports child view ordering, but Vimeflow currently renders overlays as React DOM inside the app window. Turning those into native views creates a second renderer surface.
A - Alternatives
Option A gives most of the UX benefit with lower risk. Reach for Option B only if hiding browser content under overlays becomes unacceptable.

Option C - Replace WebContentsView with the Electron webview tag reject

Render browser pages as <webview> custom elements inside React so DOM overlays and browser content share one DOM-ish layout tree.

IDEA

I - Intent
Make z-index behavior feel natural by turning the browser pane into a renderer-owned element.
D - Danger
Electron explicitly discourages new webview usage, it requires enabling webviewTag, and preload hardening becomes more sensitive.
E - Explain
The appeal is real: the host page controls layout directly. The cost is moving from a recommended API to a less stable embed path with event-routing caveats.
A - Alternatives
Option A keeps the recommended browser primitive while solving the actual coupling. Option C is not worth the regression risk.

Option D - Offscreen render browser pages into a DOM canvas research only

Render remote pages offscreen and draw their frames into a canvas or shared texture inside the BrowserPane DOM subtree.

IDEA

I - Intent
Make browser pixels fully part of the React composition layer, enabling true DOM overlays and visual effects.
D - Danger
Input routing, text selection, IME, cursor shape, accessibility, high frame-rate video, GPU memory, and audio focus all become custom product work.
E - Explain
Electron can emit offscreen frames, but that is a rendering primitive, not a complete interactive browser pane.
A - Alternatives
Keep this for future streaming or preview modes. It should not replace the durable workspace browser.

Continue with areBrowserPanesOccluded and require every overlay change to update the list.

Migration Plan

  1. Add src/features/workspace/overlays/OverlayStackProvider.tsx with descriptor registration, plane ordering, rectangle intersection, and a single scheduled recompute path.
  2. Add useOverlayRegistration() and wire it into CommandPalette, UnsavedChangesDialog, burner terminal overlays, pane rename, drag covers, and workspace banners.
  3. Add useNativeSurface() to BrowserPane. The hook should replace isOccluded with a derived nativeSurface.occluded value.
  4. Keep the existing setBrowserPaneBounds() IPC and BrowserPaneController.applyRecordBounds() behavior. The implementation can later add view.setVisible(false) if Electron testing shows it is cleaner than zero bounds.
  5. Delete areBrowserPanesOccluded after each current overlay has a registration test. Keep a temporary compatibility flag only during the migration PR if needed.

Acceptance Criteria

  • Adding a new overlay requires registering its overlay plane, not editing WorkspaceView browser-specific logic.
  • Command palette, unsaved dialog, burner popup, rename overlay, drag layer, and banners all hide native browser content when they should.
  • Browser panes remain mounted, preserve tab state, preserve CDP target identity, and restore active-tab bounds after overlays close.
  • Unit tests cover plane ordering and rect intersection. BrowserPane component tests cover bridge calls. Electron integration covers at least one DOM overlay over a real native browser pane.
