Codex 会话切换后 · Agent 状态栏停更 — 根因 + 修复方案 技术笔记 v2 · 2026-06-20 · 根因:Claude+codex 一致 方案:codex 两轮复核通过

v1 的修复方向是"后台轮询重定位";本版(v2)因为多会话开销改为按需手动 reattach,并记录 codex 第二轮复核发现的一个 macOS 关键问题。仍是说明文档,不是代码。

一句话

根因:codex 的监视器在 attach 时把 rollout 文件路径钉死,/clear / resume 会写到新文件但监视器还盯旧的 → 永远收不到新事件。
修复:不轮询(8 会话开销太大),改成按需 reattach —— 复用已有的 start_agent_watcher(它本来就是"安全原子重启"),由一个按钮 + /clear 自动触发,并把状态栏 header 标红提示"需重连"。

1 · 现象 / 复现

  • 场景 A(/clear + 继续):状态栏不再新增,一直空白。
  • 场景 B(直接 resume 另一会话):状态栏永远不再更新。
  • 对照:Claude Code 无此问题 —— codex 特有。

2 · 根本原因(已确认,未变)

监视器在 attach 时只解析一次 rollout 路径并钉死,之后不再更新:

  • attach 只解析一次 session_lifecycle.rs:843/853;路径冻结成 target_path watcher_runtime.rs:772。
  • notify 监父目录(非递归):996,但回调只认/只读旧文件 :888 / :908。
  • /clear(新线程→新 rollout)、in-session resume(另一 rollout)都在同一 PID;前端只在 PID 变化时才重启监视器 useAgentStatus.ts:299/340 → 不触发。
正常:
codex
→写→
rollout-A
→读→
watcher 盯 A ✓
→
状态栏更新
切换后:
codex
→写→
rollout-B新文件
✗ 没人读 B
watcher 仍盯 A
→
状态栏冻结

codex 只给"静态"路径提示 codex/mod.rs:111;Claude 每次状态都从 statusline 重新解析路径 claude_code/mod.rs:52 → 自愈,所以 Claude 没事。

3 · 方案演变:为什么不用自动轮询

v1 想用"后台周期性重定位",但有两个问题:

  • 多会话开销:你经常同时跑 ~8 个 codex。而正确的 re-locate 必须查"进程当前打开的 rollout 文件描述符"(open-FD,即 lsof,每次 ~10–40ms 的进程级开销)—— 单靠 sqlite 不够(codex 复核已证:多个同 cwd 行会判定"无法确定",resume 旧参数会永远胜出)。8 会话 × 每秒轮询 ≈ 持续 ~160ms/s CPU,且随会话数线性增长。控制台的本职就是跑很多 agent,这个成本画像正好相反。
  • 测量也测不准:在你多会话环境里跑测时脚本,"活跃写入"在两个后台会话间反复横跳,拿不到干净的"动作→切换"延迟。但它给了三个更有用的结论:① 环境确实多会话 → 一切必须按 pid/cwd 严格限定;② 切换是秒级(非毫秒);③ state.sqlite 被所有会话疯狂写 → 事件驱动它等于近乎持续触发,没意义。

结论:不轮询。把"昂贵但正确"的 re-locate 只在需要时执行 —— 这正是你提的手动 reattach 思路。

4 · 修复方案:按需 Reattach(codex 两轮复核通过)

核心发现(省了一大堆活):reattach 不需要新机制 —— 直接重新调用已有的 start_agent_watcher(ptyId) 即可。它走 run_watch_sequence:locate → spawn → 原子 register session_lifecycle.rs:843,而且已经是"先建新、成功后原子替换、失败保留旧"的安全重启 :811-842。重启会重放新 rollout → 先发 agent-status 事件 watcher_runtime.rs:1040 → 前端那个抑制闩自动解除 → 状态栏恢复。

1
手动按钮:在状态栏 header(就是你截图那个 "CLAUDE / updated now" 芯片 AgentStatusPanel/Header.tsx)加一个 Reattach 按钮,复用现有"刷新中"动画态。覆盖无法检测的 in-session resume。
2
/clear 自动:前端已经能检测敲入的 /clear WorkspaceView.tsx:557 → 自动 reattach。但要延迟/有限重试:检测到 /clear 时 codex 往往还没打开新 rollout,立刻 re-locate 会又抓到旧文件。
3
变红/失效态:把 header 标红表示"会话可能已变,点此重连"。复用现有刷新态机制,不需要新的 polling。(被动"兄弟 rollout 写入"提示被否决:共享日期目录下 8 会话会刷爆,除非做昂贵的 PID 过滤。)
4
前端安全:绕过 watcherStartedRef 这个"只启动一次"的闸,但复用现有 single-flight(watcherStartInFlightRef)+ generation;不要"先停再起",交给后端原子替换。

没有轮询、没有后台督导器、没有 1.5s 这种拍脑袋的间隔 —— 之前那个"间隔取多少"的问题直接消失。

⚠️ 5 · codex 复核抓到的最大风险:macOS 上 open-FD 是关的

判定新 rollout 唯一可靠的信号是"进程当前打开的 rollout FD"。但目前 open-FD 发现只走 Linux 的 /proc:default_proc_root() 在非 Linux 返回 None config.rs:128,open_rollout_paths_from_proc 也是 /proc 实现 locator.rs:513。

也就是说:在你的 macOS 上,只"重排 resolver 优先级"根本修不好 —— 必须新增一个 macOS 的 lsof open-FD provider,否则检测不到 in-session resume。这是本次修复最大的新增工作量 + 最大风险点,也正是这套"先和 codex 对齐"流程帮我们提前避开的"测试通过、到你机器上却没用"的坑。

6 · 解析器(Resolver)Option A

把本进程的 open-FD 设为权威信号(排在 resume-argv 之前),只有在"还看不到 FD"时才回退到 argv / sqlite。这样:in-session resume 能被接住;codex resume <id> 启动期(FD 还没开)仍走 argv;同一 cwd 多行已由 choose_state_candidate 安全报错 locator.rs:402/465。attach 行为不回归。

7 · 前端 latch(下游,不用动)

/clear 设的抑制闩 useAgentStatus.ts:207/526 只是后端卡死的放大效应;后端 reattach 一恢复发事件,带新 agentSessionId 的第一条就会清掉它。无需改前端 latch,只验证即可。
