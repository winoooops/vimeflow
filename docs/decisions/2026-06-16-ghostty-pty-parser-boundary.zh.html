Ghostty PTY Parser 决策：Adapter 拥有 Parser，应用消费事件

这份决策记录下一阶段 Ghostty spike 的前置架构选择：parser 归 renderer adapter 所有，但应用层只消费统一事件；PTY 输出统一成 output chunk，adapter 内部决定使用 string write 还是 byte write。

日期 2026-06-16 分支 codex/ghostty-lazy-plain-text-renderer 状态 Accepted for spike PR1

决策：采用 Option A+。Parser 留在 renderer adapter 内部，xterm adapter 继续使用 string write，Ghostty adapter 可以使用 byte write；应用层不再记忆不同 adapter 的写入机制，只把 TerminalOutputChunk 交给 parser adapter，并监听统一的 TerminalParserEvent。

01 当前链路是什么样

当前系统已经有一个比较清晰的 terminal service 边界，但输出数据仍然以 string 为主。后端 live 输出和 restore replay 都会把 PTY 读到的 bytes 通过 String::from_utf8_lossy 转成字符串。 前端再通过 pty-data 事件收到 data、offsetStart、byteLen。

PTY shell/agent 进程输出原始 bytes
Rust backend String::from_utf8_lossy 生成字符串
IPC event pty-data 携带 string + byte offset
useTerminal 写入 TerminalSurface.write(data)
Renderer parser xterm/plain-text parser 触发 OSC 7 等 handler

这里有一个很重要的好消息：offsetStart 和 byteLen 已经存在。这说明 restore/dedupe 的 cursor 已经是 byte-aware 的。坏消息是 payload 本身仍然不是 byte-preserving。

02 已接受的决策：Option A+

我们选择“parser 属于 renderer adapter，但 app 只依赖 generic parser contract”。这保留 xterm 和 Ghostty 各自 parser 的真实行为，同时避免 Body、useTerminal、session manager 这些上层代码知道 “哪个 adapter 要 string、哪个 adapter 要 bytes”。 

type TerminalOutputChunk = {
  text: string
  bytesBase64?: string
  offsetStart: number
  byteLen: number
  phase: 'live' | 'restore'
}

type TerminalParserEvent =
  | {
      type: 'cwd'
      source: 'osc7'
      uri: string
      path: string
      phase: 'live' | 'restore'
      offsetStart: number
    }

interface TerminalParserAdapter {
  readonly id: string
  readonly capabilities: {
    acceptsText: boolean
    acceptsBytes: boolean
  }

  writeOutput(chunk: TerminalOutputChunk): void
  onEvent(handler: (event: TerminalParserEvent) => void): TerminalDisposable
}

边界要求

  • xterm adapter 从 TerminalOutputChunk.text 写入 xterm parser。
  • Ghostty adapter 优先从 TerminalOutputChunk.bytesBase64 还原 bytes，再写入 Ghostty/libghostty-vt parser。
  • 上层只处理 TerminalParserEvent，不直接注册 registerOscHandler(7)。
  • Ghostty spike 必须继续挂在显式 feature flag / renderer selector 后面。

2. Provider 是选择器 Provider 负责把一个 driver factory 交给 Ghostty adapter。它不消费 PTY，也不直接发 cwd 事件。
3. Driver 是状态拥有者 Milestone 2 要接入的真实 driver 应该拥有 terminal state，而不是把 screen/cursor/OSC 逻辑散回应用层。
4. OSC 7 走旧数据路径 Driver 可以发现 cwd，但必须通过 adapter 包装成 TerminalParserEvent，继续保护 pane cwd 和 agent observability 的分离。

03 为什么 Ghostty 需要认真对待 bytes

Lossy decode

无效 UTF-8 会被替换

from_utf8_lossy 会把非法字节替换成 U+FFFD。 对 xterm 当前字符串路径来说通常够用，但对一个严肃的 terminal parser 来说，这已经丢失了输入事实。

Chunk boundary

多字节字符可能跨 read 边界

PTY read 的切分不等于 UTF-8 字符边界，也不等于 escape sequence 边界。逐 chunk lossy decode 会让 parser 无法还原原始流。

Native parser

Ghostty/libghostty-vt 更像 byte parser

如果 spike 的目标是评估 Ghostty 的 terminal state machine， 它应该尽量吃原始 bytes，而不是吃已经被前端协议塑形过的字符串。

工程判断

当前 string pipe 可以继续服务 xterm 默认路径，但 Ghostty spike 应该被视为一个新的 transport contract 验证点。也就是说： xterm 可以暂时从 byte chunk 解码成 string， 但 ghostty adapter 不应该被迫接受 lossy string。

04 Parser 到底归谁管

现在应用层最明显的 parser 依赖是 OSC 7：renderer 实例暴露 parser.registerOscHandler(7, ...)，Body.tsx 在 handler 中解析 cwd、做 stale guard、restore suppression，并更新 pane 的 cwd 状态。

这说明 parser 不只是“渲染内部细节”。它已经在为应用层提供语义事件。 如果 Ghostty 也有自己的 parser，那么我们必须回答： 应用层应该监听谁产生的 OSC/cwd/title/hyperlink 事件？

责任 现在的位置 Ghostty 风险
字节流解析 xterm 或 plain-text renderer 内部 不同 renderer 可能对同一 stream 产生不同语义
OSC 7 cwd 提取 Body.tsx 注册 parser handler 后处理 restore replay 和 live output 容易混在一起
Viewport/test buffer TerminalViewportReader contract Ghostty 的 viewport 需要映射到现有 E2E 读取契约
渲染 xterm addon/canvas/DOM 或 plain-text DOM Ghostty 可能需要 native surface，而不是普通 DOM writer

05 OSC 7 与 agent observability 的真实关系

这里必须分清两个 cwd 通道。它们都重要，但语义不同，不能在 Ghostty spike 中混成一个状态源。

通道 来源 当前消费者 必须保护的结构
Pane live cwd Terminal output 中的 OSC 7，以及 agent text hint fallback Body.tsx → onCwdChange → updatePaneCwd → backend PTY cwd cache Session.panes[].cwd、update_session_cwd、 git header、burner align、workspace restore
Agent observed cwd Backend transcript watcher 发出的 agent-cwd event useAgentStatus 和 agent status panel AgentStatus.cwd，来自 Claude/Codex transcript 的结构化 cwd/workdir

OSC 7 优先级

PR1 的最高优先级是保护 pane cwd 的现有数据路径。新的 parser event 只是把 registerOscHandler(7) 变成 generic { type: 'cwd', source: 'osc7' } 事件；它不能改变 Session.panes[].cwd 的存储位置，也不能替代 agent-cwd 这个 observability 通道。

06 方案回顾

A

接受：Parser 留在 renderer adapter

xterm 用 xterm parser，Ghostty 用 Ghostty parser。adapter 负责把 OSC/cwd/title 等事件翻译成统一的前端事件。

这就是我们接受的方向，但补上 output chunk 和 parser event contract， 所以更准确地叫 Option A+。

B

Parser 下沉到 backend/native

后端保留原始 bytes，并由一个 native parser 统一产出语义事件。 前端 renderer 只负责画面。

优点是语义来源单一，restore/cache 更可控。缺点是会和 xterm 内部 parser 重复，短期改动面也最大。

C

并入 A+：先做 split contract

先定义 byte transport 和 parser event contract。parser 可以暂时由 renderer adapter 提供，但应用层只消费统一事件。

这个 contract 不作为单独方案推进，而是作为 Option A 的必要护栏。

07 建议的 Ghostty spike 路线

  1. 定义输出包：增加类似 TerminalOutputChunk 的结构，至少包含 offsetStart、byteLen、当前兼容用 text，以及实验用 bytesBase64 或等价 byte payload。
  2. 保留兼容层：xterm adapter 继续走 string write， 但它应该从统一 output chunk 中取 text。这样默认行为不变。
  3. 建立 parser event bus：把 OSC 7 这类语义抽成 TerminalParserEvent，例如 { type: 'cwd', source: 'osc7', path, uri, phase }。 phase 必须区分 live 和 restore replay。
  4. 实现 Ghostty adapter spike：只在 VITE_TERMINAL_RENDERER=ghostty 之类的 opt-in 路径启用。 第一版目标不是替换 UI，而是验证 bytes 能进入 Ghostty parser，并能产出 viewport snapshot 与 parser events。
  5. 补 contract tests：用跨 chunk OSC、多字节 UTF-8、非法 bytes、 restore replay OSC 这几类 fixture 证明新边界可靠。

08 术语与开放问题

  • byte-preserving 指传输层不丢原始 bytes。PTY 读到什么 bytes，Ghostty parser 就能拿到同样的 bytes。 这是 PR1 需要保护的能力。
  • byte-perfect 更强，表示最终 terminal state、viewport、 restore、render 都和真实 terminal 完全一致。Ghostty spike 第一阶段不承诺这个，只先保证 parser 输入 byte-preserving。
  • bytesBase64 只是 IPC 初版方案，不是 terminal 语义。 因为当前 IPC event 是 JSON/string，base64 是最小改动的 raw bytes 容器。它有性能成本；如果 Ghostty 路径高吞吐有问题，再换 binary transport。
  • restore replay 指 app 重新 attach 一个已有 PTY 时，把历史输出写回 terminal 以恢复 scrollback。历史输出里可能包含旧 OSC 7；所以 parser event 必须携带 phase: 'restore'，避免旧 cwd 污染 live pane cwd。
  • OSC 7/cwd 第一阶段由 adapter 发 normalized event，不下沉到 backend。 这样可以保护 xterm 默认路径，并让 Ghostty spike 独立验证自己的 parser。
  • Ghostty adapter 的最小成功标准是什么？建议定义为：能吃 raw bytes、能产出 cwd event、能提供 viewport reader、不会影响默认 xterm bundle。
