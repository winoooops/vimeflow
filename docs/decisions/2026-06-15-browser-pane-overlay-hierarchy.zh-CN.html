Vimeflow technical note

让 BrowserPane 进入同一套 overlay hierarchy

现在的 browser pane 通过一个 workspace-level boolean 去手动列出每个 overlay source。这个方式能跑，但会让 BrowserPane 变成特殊路径。 更好的形态是：让 overlay source 自己注册，让 BrowserPane 像其他 terminal pane 一样订阅同一套 hierarchy。

Short Answer

可以让 BrowserPane 共享 terminal panes 的 overlay hierarchy，但不能 只靠 CSS z-index。Electron 的 WebContentsView 不是 React DOM node。React overlay 想压过它，必须通过 main process 去 hide、move、reorder native view，或者把 overlay 自己也做成 native surface。

推荐方案：继续使用 WebContentsView，删除手写的 areBrowserPanesOccluded list，引入 workspace-level OverlayStackProvider。每个 overlay primitive 注册自己的 plane 和 native-view policy；BrowserPane 注册自己的 native content rect，并从这个 shared stack 派生 occlusion。

Current Shape

当前 BrowserPane 已经在 split-pane model 里，但它的 native page area 由 parent 计算出来的 flag 控制：WorkspaceView 组装 areBrowserPanesOccluded，TerminalZone 传给 SplitView，SplitView 再传给 BrowserPane 作为 isOccluded。 

// src/features/workspace/WorkspaceView.tsx
const areBrowserPanesOccluded =
  terminalFitDeferred ||
  showUnsavedDialog ||
  commandPalette.state.isOpen ||
  hasVisibleBurner ||
  paneRenameNode !== null ||
  fileError !== null ||
  infoMessage !== null

// src/features/browser/components/BrowserPane.tsx
const visible =
  isActiveRef.current &&
  !isOccludedRef.current &&
  rect.width > 0 &&
  rect.height > 0

void setBrowserPaneBounds({ sessionId, paneId, bounds, visible })

Electron controller 会把 visible: false 映射到 zero-size native bounds。这个 hiding mechanism 本身是对的，问题是 source of truth 离 overlay system 太远。 

// electron/browser-pane.ts
private applyRecordBounds(record: BrowserPaneRecord): void {
  const active = this.activeTab(record)
  for (const tab of record.tabs.values()) {
    tab.view.setBounds(
      tab === active
        ? visibleBounds(record.lastBounds, record.visible)
        : { x: 0, y: 0, width: 0, height: 0 }
    )
  }
}

问题不是 visible 机制，而是每新增一个 dialog、popup、 rename surface、banner、drag layer 或 future overlay，都必须记得更新 browser-specific boolean。这个 hidden coupling 会让 workspace overlay hierarchy 和 BrowserPane 互相绑死。

Electron Facts

Fact Implication for Vimeflow
WebContentsView 是 main-process native View，不是 DOM node。 React portal 和 Tailwind z-index 不能直接压过它。renderer 和 main process 必须一起协调 bounds 或 visibility。
View.addChildView(view, index) 可以调整 native child view order，重新 add existing view 也会让它成为 topmost。 Native overlay 技术上可行，但前提是 overlay 自己也有 native view/window representation。
View.setVisible(false) 存在，bounds 也可以设置为 0x0。 现有 zero-bounds strategy 可以保留，但应封装到 native-surface contract 后面。
Electron 不建议新功能优先使用 <webview>。 为了让 z-index 更自然而切到 DOM custom element，会把稳定性和 security risk 带回来。
Offscreen rendering 可以输出 bitmap 或 shared texture。 它能让 browser pixels 进入 DOM/canvas hierarchy，但 input、IME、 accessibility、video、performance 都会变成自研问题。

Target Contract

BrowserPane 不应该由 WorkspaceView 告诉它「现在有 overlay」。它应该参与 terminal canvas 已经使用的同一套 overlay hierarchy。

Overlay sources self-register

Dialog、popover、drag cover、command palette、rename controls 和 banner 都声明自己的 overlay plane，以及是否会 occlude native browser surface。

Native surfaces subscribe

BrowserPane 把 page rect 注册为 native surface。当任何 higher overlay plane 与它 intersect，或全局覆盖它时，就收到 occluded: true。

Proposed Type Shape

type OverlayPlane =
  | 'pane-chrome'
  | 'popover'
  | 'dialog'
  | 'palette'
  | 'drag'
  | 'toast'

type NativeOcclusionPolicy = 'none' | 'intersects' | 'global'

interface OverlayDescriptor {
  id: string
  plane: OverlayPlane
  isOpen: boolean
  nativeOcclusion: NativeOcclusionPolicy
  getRect?: () => DOMRectReadOnly | null
}

interface NativeSurfaceDescriptor {
  id: string
  owner: 'browser-pane'
  getRect: () => DOMRectReadOnly | null
  belowPlane: OverlayPlane
}

Occlusion Sequence

CommandPalette opens
  -> useOverlayRegistration({ plane: 'palette', nativeOcclusion: 'global' })
  -> OverlayStackProvider recomputes native surface visibility
  -> BrowserPane receives occluded = true
  -> BrowserPane sends setBrowserPaneBounds({ visible: false })
  -> BrowserPaneController applies zero bounds or setVisible(false)
  -> React palette is now visually and interactively topmost

CommandPalette closes
  -> overlay descriptor is removed
  -> BrowserPane receives occluded = false
  -> BrowserPane sends latest measured bounds with visible = true
  -> controller restores active tab bounds and focus policy decides refocus

BrowserPane Pseudocode

export const BrowserPane = (props: BrowserPaneProps): ReactElement => {
  const contentRef = useRef<HTMLDivElement>(null)
  const nativeSurface = useNativeSurface({
    id: `browser:${props.session.id}:${props.pane.id}`,
    owner: 'browser-pane',
    getRect: () => contentRef.current?.getBoundingClientRect() ?? null,
    belowPlane: 'pane-chrome',
  })

  const syncBounds = useCallback((): void => {
    const rect = contentRef.current?.getBoundingClientRect()
    if (!rect || !nativePaneReadyRef.current) {
      return
    }

    void setBrowserPaneBounds({
      sessionId: props.session.id,
      paneId: props.pane.id,
      bounds: rectToBounds(rect),
      visible:
        props.isActive &&
        props.pane.active &&
        !nativeSurface.occluded &&
        rect.width > 0 &&
        rect.height > 0,
      shortcutContext: shortcutContextRef.current,
    })
  }, [props.isActive, props.pane.active, props.pane.id, nativeSurface.occluded])

  useLayoutEffect(syncBounds)

  return <div ref={contentRef} data-testid="browser-pane-content" />
}

Overlay Source Pseudocode

export const CommandPalette = ({ state, ...props }: Props): ReactElement => {
  const panelRef = useRef<HTMLDivElement>(null)

  useOverlayRegistration({
    id: 'command-palette',
    plane: 'palette',
    isOpen: state.isOpen,
    nativeOcclusion: 'global',
    getRect: () => panelRef.current?.getBoundingClientRect() ?? null,
  })

  return state.isOpen ? (
    <div className="fixed inset-0 z-100">
      <div ref={panelRef}>...</div>
    </div>
  ) : null
}

Technical Difficulties

Difficulty Why it matters Mitigation
DOM compositor 和 native view compositor 不同 即使 React overlay 的 z-index 更高，它仍可能被 native browser page 盖住。 把 native browser content 当作 registered surface，通过 main process 显式控制 visibility。
Overlay ownership 很分散 Command palette、dialog、burner terminal、rename affordance、banner 和 drag cover 分布在不同组件里。 每个 overlay 自己注册，不让 parent 手动维护完整 list。
Rect measurement 容易漂移 ResizeObserver 能捕捉 size change，但纯 position change 仍需要 render-time sync。 保留现有 post-render syncBounds() pattern，增加 shared requestAnimationFrame scheduler，并继续 de-dupe bounds key。
Focus return 可能抢用户意图 Dialog 关闭后，browser page 可能重新拿 focus，而用户其实期望回到 terminal zone、dock 或 command palette workflow。 保留 BrowserPane 的 focus guards。只有在 pane 本来 active 且没有 higher-priority focus target 时，才 refocus native content。
Pointer capture during drag Native web content 可能吞掉 split handle、dock handle 或 sidebar handle 的 mouse events。 把 drag layer 注册为 nativeOcclusion: 'global'，不要让 每个 drag source 调 browser-specific API。
一个 BrowserPane 有多个 native tabs 只有 active native tab 应该拿到真实 bounds；inactive tab views 要保持 alive 但 hidden。 继续让 BrowserPaneController.applyRecordBounds() 作为 active-tab projection 的 authority。
Security boundary 不能放松 Remote pages 是 untrusted。更简单的 DOM embed 可能重新引入 Node、 preload 或 navigation risk。 保留 sandboxed WebContentsView： nodeIntegration: false、 contextIsolation: true，并保留现有 navigation/popup policies。

Alternatives Analysis

Option A - Overlay registry with BrowserPane native-surface subscription recommended

保留当前 WebContentsView browser architecture，但把 parent flag 换成 generic overlay/native-surface registry。

IDEA

I - Intent
让 BrowserPane 共享 workspace overlay hierarchy，不再要求每个 overlay source 记住 browser-specific gate。
D - Danger
Registry 会变成 infrastructure；如果某个 overlay primitive 忘记注册 descriptor，native occlusion 仍可能漂移。
E - Explain
这个方案保留 working native browser surface，并把 coupling 收敛到一个 explicit contract：overlay 描述自己的 plane，native pane 订阅它。
A - Alternatives
只有在 Vimeflow 需要 native overlay 盖在 native page 上且 page 仍可见时，才考虑 Option B。Option E 只能作为短期 bridge。

const occluded = overlays.some((overlay) =>

Option B - Native overlay compositor in Electron main powerful

把 app shell 和 overlay surfaces 移到 native BaseWindow view hierarchy，或者增加一个 dedicated overlay WebContentsView 放在 browser page views 上方。

IDEA

I - Intent
让 overlay 真正成为 browser native view 上方的 native sibling， translucent glass surface 下方可以继续看到 browser page。
D - Danger
这是 windowing architecture change。风险包括 focus bugs、 lifecycle leaks、cross-platform child-window behavior，以及更大的 test matrix。
E - Explain
Electron 支持 child view ordering，但 Vimeflow 现在的 overlays 是 app window 内的 React DOM。把它们做成 native views 会引入第二个 renderer surface。
A - Alternatives
Option A 用更低风险拿到大部分 UX 收益。只有在「overlay 下必须保留 browser visible」成为硬需求时，才值得走 Option B。

Option C - Replace WebContentsView with Electron webview tag reject

把 browser pages 渲染成 React 里的 <webview> custom element，让 DOM overlay 和 browser content 看起来共享同一套 layout tree。

IDEA

I - Intent
通过 renderer-owned element 让 z-index behavior 更自然。
D - Danger
Electron 明确建议优先考虑 alternatives；它还要求启用 webviewTag，preload hardening 也会更敏感。
E - Explain
吸引力很明显：host page 可以直接控制 layout。但代价是从推荐 API 切到更脆弱的 embed path。
A - Alternatives
Option A 保留推荐的 browser primitive，同时解决真正的 coupling。 Option C 不值得承担 regression risk。 
const BrowserPaneWebview = ({ url }: Props): ReactElement => (

)

Option D - Offscreen render browser pages into DOM canvas research only

把 remote pages offscreen render，然后把 frames draw 到 BrowserPane DOM subtree 的 canvas 或 shared texture。

IDEA

I - Intent
让 browser pixels 完全进入 React composition layer，支持真正的 DOM overlay 和 visual effects。
D - Danger
Input routing、text selection、IME、cursor shape、accessibility、 high frame-rate video、GPU memory、audio focus 都会变成自研成本。
E - Explain
Electron 可以 emit offscreen frames，但这只是 rendering primitive， 不是完整 interactive browser pane。
A - Alternatives
可以留作 streaming/preview mode 的 future research，不应该替代 durable workspace browser。 
const win = new BrowserWindow({

Option E - Keep the parent boolean and document the overlay list temporary

继续使用 areBrowserPanesOccluded，并要求每个 overlay change 都更新这个 list。

IDEA

I - Intent
避免新增 infrastructure，同时保持当前 browser overlays 可用。
D - Danger
下一个 overlay 很容易忘记加 flag，从而 ship 一个真实 visual/input bug。 Coupling 仍然藏在 WorkspaceView 里。
E - Explain
作为 spike path 它是合理的，因为它验证了 native hiding mechanism，并保持 BrowserPane 稳定。
A - Alternatives
只能作为 Option A 落地期间的 compatibility fallback，不应该成为长期 contract。

Migration Plan

  1. 新增 src/features/workspace/overlays/OverlayStackProvider.tsx： 支持 descriptor registration、plane ordering、rectangle intersection 和 shared scheduled recompute path。
  2. 新增 useOverlayRegistration()，并接入 CommandPalette、UnsavedChangesDialog、 burner terminal overlays、pane rename、drag covers、workspace banners。
  3. 新增 useNativeSurface() 给 BrowserPane。 这个 hook 用 derived nativeSurface.occluded 替换 isOccluded prop。
  4. 保留现有 setBrowserPaneBounds() IPC 和 BrowserPaneController.applyRecordBounds()。后续如果 Electron integration test 证明 view.setVisible(false) 更干净，再把 implementation 从 zero bounds 换过去。
  5. 当当前 overlay 都有 registration test 后，删除 areBrowserPanesOccluded。如果 migration PR 需要，可以短期保留 compatibility flag。

Acceptance Criteria

  • 新增 overlay 只需要注册 overlay plane，不需要编辑 WorkspaceView 里的 browser-specific logic。
  • Command palette、unsaved dialog、burner popup、rename overlay、drag layer 和 banners 都能在需要时 hide native browser content。
  • Browser panes 保持 mounted，保留 tab state，保留 CDP target identity， overlay 关闭后恢复 active-tab bounds。
  • Unit tests 覆盖 plane ordering 和 rect intersection；BrowserPane component tests 覆盖 bridge calls；Electron integration 至少覆盖一个 DOM overlay over real native browser pane 的场景。
