Technical Note

Pane Layout 增加 3x2 的技术方案

这份说明聚焦 terminal zone 里的 pane layout 体系，先把当前结构、 真实耦合点、以及引入 3x2 时会撞到的限制梳理清楚。 结论很直接：现在加一个新 pill 很容易，但如果目标是一个真正可交互、可 resize、 可 restore、未来还可继续扩展的 layout 系统，就不该继续把 layout 元数据、 autoshrink 规则、divider 逻辑、ratio 结构散落在不同模块里。

范围：top chrome + terminal zone 日期：2026-06-17 建议 id：grid3x2 目标容量：6 panes
当前状态 5 个 layout single / vsplit / hsplit / threeRight / quad
核心问题 规则分散 layout id、文案、grid、shortcut、autoshrink 分别藏在多处
真正阻塞点 ratio 模型过窄 现在只支持一条列分割和一条行分割，不够支撑 3x2

现状结构

UI 入口

Layout pill group 在 top chrome

WorkspaceView.tsx 里渲染 LayoutSwitcher。它直接遍历 Object.values(LAYOUTS)，所以只要往 LAYOUTS 里加一个 layout，新的 pill 就会出现。

状态归属

layout 属于 Session

每个 session 都有一个 layout: LayoutId，由 useSessionManager.setSessionLayout() 修改。 所以 layout 不是纯视觉状态，而是 session shape 的一部分。

渲染路径

TerminalZone -> SplitView

TerminalZone 每个 session 渲染一个 SplitView，隐藏 inactive session 但不卸载它们， 以保持 PTY 活着。SplitView 再根据 session.layout 选择 grid、slot、divider。

隐式中心

现在的“中心”其实是散的

layouts.ts 只是 layout 元数据的一部分来源， 但 autoshrink 在 paneLifecycle.ts， shortcut cycle 在 usePaneShortcuts.ts， restore whitelist 在 groupSessionsFromInfos.ts， divider 行为在 SplitDividers.tsx。这不是一个真正的中心注册表。

你提出的方向是对的：需要中心 registry / controller

推荐方案

不要只保留一个常量表，应该升级成 layout registry

这个 registry 不一定非得是 class；可以是一个深一点的模块， 但职责上要像一个 controller / registry。关键不是语法形式， 而是要把“layout 的所有规范性知识”集中起来。 

interface LayoutSpec {
  id: LayoutId
  label: string
  capacity: number
  icon: 'single' | 'vsplit' | 'hsplit' | 'threeRight' | 'quad' | 'grid3x2'
  cycleOrder: number
  resolveGrid: (ratios: LayoutRatiosModel) => ResolvedGrid
  createDefaultRatios: () => LayoutRatiosModel
  getDividerSpecs: (ratios: LayoutRatiosModel) => DividerSpec[]
  autoShrinkTo: (nextPaneCount: number, ctx: AutoShrinkContext) => LayoutId
}
应该收拢进 registry 的东西
  • LayoutId 与 label 文案
  • pill 排序与 shortcut cycle 顺序
  • capacity
  • grid area mapping
  • 默认 ratio
  • divider 清单与 divider 绑定的 axis / css var
  • autoshrink 规则
  • restore 时的 layout 合法性判断

重点：现在很多逻辑是“layout id 被传到各处，然后各处自己 switch 一遍”。 这会让每次新增 layout 都变成全仓库补 case。registry 的价值就是把这些 switch 收口。

当前耦合点

模块 当前假设 问题
src/features/sessions/types/index.ts LayoutId 固定为 5 个值 新增 layout 要先改 union，再触发一串编译/测试扩散
layouts.ts capacity 只允许 1 | 2 | 3 | 4 3x2 直接超出类型边界
usePaneShortcuts.ts shortcut cycle 依赖 LAYOUTS 插入顺序 一旦插入新 layout，用户的 Cmd/Ctrl+\ 行为会被静默改写
paneLifecycle.ts autoshrink 只理解 1 / 2 / 3 / 4+ 从 6 pane 关到 5 pane、再关到 4 pane 时没有清晰策略
groupSessionsFromInfos.ts restore 只接受现有 5 个 layout id 新 layout 落盘后，旧 restore 路径会退回 single
SplitDividers.tsx divider 组件是按 layout 写死分支 扩 layout 时要继续复制 case，系统只会越来越脆

更大的问题：LayoutRatios 不该再只服务 2x2 / 3x2

当前模型 
interface LayoutRatios {
  col: number
  row: number
}

这个模型默认“每个 layout 最多只有一条垂直分割线和一条水平分割线”。 它对 vsplit、hsplit、threeRight、 quad 够用，但只要遇到 3 列以上布局，就天然不够。

推荐模型 
interface LayoutRatiosModel {
  cols: number[]
  rows: number[]
}

这里的语义应当是“pane track 比例”，不是“divider 的位置快照”。 例如：

  • single - { cols: [1], rows: [1] }
  • vsplit - { cols: [1, 1], rows: [1] }
  • hsplit - { cols: [1], rows: [1, 1] }
  • quad - { cols: [1, 1], rows: [1, 1] }
  • grid3x2 - { cols: [1, 1, 1], rows: [1, 1] }

关键收益：以后新增 4x2、3x3、 “主区 + 右侧 3 stack” 这类 layout，就不需要先重做 ratio 类型。 layout 的几何复杂度由 spec 决定，不再被一个固定的 { col, row } 结构卡死。

ratio 模型如何落地

新的思路

用 track 数组，而不是单个分割比例

现在 resolveGrid() 拼的是 --split-col / --split-col-end 这种二段式模板。 新模型应该改成：给定 cols / rows track 数组，生成完整的 grid-template-columns 和 grid-template-rows。 

cols: [1, 1, 1]
-> "var(--col-0,1fr) 8px var(--col-1,1fr) 8px var(--col-2,1fr)"

rows: [1, 1]
-> "var(--row-0,1fr) 8px var(--row-1,1fr)"
divider 也要变成数据驱动 
interface DividerSpec {
  id: string
  axis: 'horizontal' | 'vertical'
  gridArea: string
  trackIndex: number
}

对 grid3x2 来说，会有两条逻辑列 divider 和一条逻辑行 divider， 但因为横向整条 bar 会切断竖线，最终渲染层面可能是 5 个 handle： 上两段竖线、下两段竖线、以及中间一条横线。

对 3x2 layout 的建议定义

layout id

grid3x2

这个命名比纯数字或模糊文案更稳。它表达的是几何结构，而不是实现细节。

grid area 建议 
areas: [
  ['p0', 'vdiv0a', 'p1', 'vdiv1a', 'p2'],
  ['hdiv', 'hdiv', 'hdiv', 'hdiv', 'hdiv'],
  ['p3', 'vdiv0b', 'p4', 'vdiv1b', 'p5'],
]

这里显式把竖线切成上下两段，避免像现在 quad 那样需要组件里手动“一个逻辑 divider，渲染成两段”的特殊处理。 这更利于泛化。

toolbar 尺寸
沿用现有 glyph 风格

autoshrink 也应注入到 registry，而不是散落在 paneLifecycle

当前问题

autoShrinkLayoutFor() 现在是一个全局函数， 用 pane count 和 current layout 直接推下一个 layout。 这在 layout 数量少时还能撑住，但 layout 一多，规则就会失控。

建议

autoshrink 应该变成每个 layout spec 的一部分，或者由 registry 统一调度。例如： 

registry.getAutoShrinkTarget({
  currentLayoutId,
  nextPaneCount,
  previousLayoutId,
})

我的建议： grid3x2 在 6 和 5 pane 时都保留自身； 到 4 pane 降为 quad； 3 pane 降为 threeRight； 2 pane 降为 vsplit，但如果当前就是纯纵向栈式语义，也可以保留 hsplit。

推荐重构顺序

阶段 目标 原因
1 引入中心 layout registry 先把 layout 的定义收口，不然后面每加一个 layout 还是到处补 switch
2 把 LayoutRatios 改成 track 数组模型 这是未来扩展的根基，不改这一层，3x2 只是硬塞进去
3 让 resolveGrid / SplitDividers 数据驱动 避免继续按 layout 写死组件分支
4 加入 grid3x2 spec、glyph、pill 此时再加 layout，代价最低，也最干净
5 补 restore / shortcut / autoshrink / tests 把行为面收完整，防止 UI 出来但系统语义不闭合

结论

你提的两个判断都对。

  • 第一，layout 体系确实需要一个更中心化的 registry / controller， 把 LayoutId、layout 文案、mapping、autoshrink、 divider 规则、shortcut 顺序统一收口。
  • 第二，LayoutRatios 不能再继续假设世界只有 { col, row } 这两个值。应该升级成可表达任意列轨道、 任意行轨道的 ratio / track 模型，才能让后续 layout 扩展是“自然增加 spec”， 而不是“每次都改底层结构”。

所以如果下一步要实现，我建议不是“先把 3x2 按 case 塞进去”， 而是先做一轮小型 layout infrastructure 重构，再上新 layout。
