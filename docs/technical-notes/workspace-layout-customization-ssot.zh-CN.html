Technical Note

Workspace Pane Layout 自定义：先定唯一数据结构，再画配置面板

目标不是再加几个预设按钮，而是让用户能创建终端形状、调整布局，并决定 workspace 使用什么布局。关键决策是：runtime、restore、divider、 ratio、icon、customization plane 都必须从同一个 PaneLayoutDefinition 派生，不能让配置页成为另一套事实来源。

SSoT：Track + Slot Rect 支持：2x3 / 3x2 / 3x3 / 4x4 支持：Main + N Stack Custom UI 只编辑 Draft
核心原则 一个 layout 定义 所有渲染、校验、持久化都从它推导
不做什么 不存 CSS 字符串 CSS grid、divider、glyph 都是派生产物
第一版边界 Rectilinear grid 矩形 slot、无重叠、默认无洞

结论：合法 pane layout 的唯一数据结构

推荐把 layout 表达为“行列轨道 + slot 矩形”。这比 grid-template-areas 更适合作为 domain model，因为它能直接表达 resize ratio、slot 边界、capacity、pane placement，也能稳定序列化。 

type PaneLayoutId = BuiltinLayoutId | `custom:${string}`
type LayoutSlotId = `slot:${string}`

interface PaneLayoutDefinition {
  schemaVersion: 1
  id: PaneLayoutId
  title: string
  source: 'builtin' | 'workspace'
  tracks: {
    columns: readonly TrackSpec[]
    rows: readonly TrackSpec[]
  }
  slots: readonly PaneSlotSpec[]
  addOrder: readonly LayoutSlotId[]
}

interface TrackSpec {
  id: string
  units: number
  minPx?: number
}

interface PaneSlotSpec {
  id: LayoutSlotId
  rect: {
    col: number
    row: number
    colSpan: number
    rowSpan: number
  }
  accepts?: readonly PaneKind[]
}

units 是给用户和配置面板看的规格单位。runtime 可以直接把它当 fr 权重使用，也可以 normalize 成 ratio；关键是 UI 不再暴露 难以一眼理解的小数比例。

Step 1

先定义 tracks：有几列、几行、比例是多少

tracks 是 resize ratio 的来源。不要用小数 ratio 当心智模型； 更推荐像 Tailwind / Bootstrap grid 一样， 把总宽度或总高度拆成 24 份。比如左侧 main 占 16/24，右侧 stack 占 8/24； 三行默认一样高，就各占 8/24。 

tracks: {
  columns: [
    { id: 'main', units: 16 },
    { id: 'side', units: 8 },
  ],
  rows: [
    { id: 'top', units: 8 },
    { id: 'middle', units: 8 },
    { id: 'bottom', units: 8 },
  ],
}
Step 2

再定义 slots：每个 pane 形状占哪里

slot rect 是 layout 的核心。它不关心当前里面有没有 terminal，也不关心 terminal 的 PTY id。它只描述“这个位置可以放一个 pane”。 

slots: [
  {
    id: 'slot:main',
    rect: { col: 0, row: 0, colSpan: 1, rowSpan: 3 },
  },
  {
    id: 'slot:side-top',
    rect: { col: 1, row: 0, colSpan: 1, rowSpan: 1 },
  },
  {
    id: 'slot:side-middle',
    rect: { col: 1, row: 1, colSpan: 1, rowSpan: 1 },
  },
  {
    id: 'slot:side-bottom',
    rect: { col: 1, row: 2, colSpan: 1, rowSpan: 1 },
  },
]
字段 在这个例子里的值 它回答的问题
tracks.columns [main, side] 横向有几块？每块宽度比例是多少？
tracks.rows [top, middle, bottom] 纵向有几块？每块高度比例是多少？
slot:main.rect { col: 0, row: 0, colSpan: 1, rowSpan: 3 } main pane 从左上开始，占整条左列
slot:side-middle.rect { col: 1, row: 1, colSpan: 1, rowSpan: 1 } 第二个 sub pane 在右列中间
addOrder ['slot:main', 'slot:side-top', 'slot:side-middle', 'slot:side-bottom'] 用户点 Add Pane 时，新的 pane 按什么顺序填空位
Step 3

runtime 再从它派生 CSS grid

注意这里是“派生”。我们不把这段 CSS 当成 source of truth， 因为它只是浏览器需要的输出格式。 

grid-template-columns:
  minmax(0, 16fr) 8px minmax(0, 8fr)

grid-template-rows:
  minmax(0, 8fr) 8px minmax(0, 8fr) 8px minmax(0, 8fr)

grid-template-areas:
  "slot-main vdiv slot-side-top"
  "slot-main vdiv hdiv-side-0"
  "slot-main vdiv slot-side-middle"
  "slot-main vdiv hdiv-side-1"
  "slot-main vdiv slot-side-bottom"
Step 4

divider 也从 slot 边界派生

左右之间有一条完整竖向边界，所以生成一个 vertical divider。右侧三个 sub pane 之间有两条水平边界，所以生成两个 horizontal divider。 

derived dividers:
[
  { axis: 'columns', trackIndex: 0 },
  { axis: 'rows', trackIndex: 0, segment: 'right-column' },
  { axis: 'rows', trackIndex: 1, segment: 'right-column' },
]

这就是为什么 divider 不应该手写在每个 layout case 里。只要 slot rect 合法，divider 可以被统一算法算出来。

同一个模型的另一个形状：Main + bottom 3 panes

如果用户想要“上面一个大 main workspace，底部三个 workspace 横向排列”， 数据结构不用换，只是 tracks 和 slot rect 换一组。这里变成 3 列、2 行：main slot 横跨上方三列， 底部三个 sub slot 各占一列。 

Main + bottom 3 panes 的逻辑网格：

columns: [left, center, right]
rows:    [main, bottom]

视觉结果：

+-------------+-------------+-------------+
|                                       |
|              main pane                |
|                                       |
+-------------+-------------+-------------+
| sub pane 1  | sub pane 2  | sub pane 3  |
+-------------+-------------+-------------+

这个例子说明：rowSpan 和 colSpan 是同一种能力的两个方向。右侧 stack 例子是 main 纵向跨 3 行； 底部横排例子是 main 横向跨 3 列。

Tracks

三列两行

columns 是三个底部 pane 的横向宽度来源；rows 是 main 区域和底部区域的高度比例。 这里同样按 24 份理解：底部三列各 8/24， 上方 main 高度 18/24，底部区域高度 6/24。 

tracks: {
  columns: [
    { id: 'left', units: 8 },
    { id: 'center', units: 8 },
    { id: 'right', units: 8 },
  ],
  rows: [
    { id: 'main', units: 18 },
    { id: 'bottom', units: 6 },
  ],
}
Slots

main 横跨三列

main slot 从第 0 列第 0 行开始，横跨 3 列、只占 1 行。底部三个 slot 在第 1 行分别占第 0、1、2 列。 

slots: [
  {
    id: 'slot:main',
    rect: { col: 0, row: 0, colSpan: 3, rowSpan: 1 },
  },
  {
    id: 'slot:bottom-left',
    rect: { col: 0, row: 1, colSpan: 1, rowSpan: 1 },
  },
  {
    id: 'slot:bottom-center',
    rect: { col: 1, row: 1, colSpan: 1, rowSpan: 1 },
  },
  {
    id: 'slot:bottom-right',
    rect: { col: 2, row: 1, colSpan: 1, rowSpan: 1 },
  },
]

对比：右侧 stack 和底部横排都不是新 layout 类型。 它们都是同一个 PaneLayoutDefinition schema 的不同数据。 这就是为什么我们要把 SSoT 定成 rect，而不是枚举一堆 mainRight3、mainBottom3 特殊 case。

一句话理解：layout definition 描述的是“地图”； session placement 描述的是“哪个 pane 住在哪个房间”；CSS grid 和 divider 是把这张地图画到屏幕上的方式，不是地图本身。

Workspace / Session 应该存什么

这里要拆清楚两种 layout 来源：app 自带一组 prebuilt layouts；用户可以在当前 workspace/session 里创建自己的 customized layouts。运行时的 layout list 不是只看代码里的预设，也不是只看用户数据，而是把两者合并： [...prebuilt, ...customized]。 

const availableLayouts = [
  ...PREBUILT_PANE_LAYOUTS,
  ...workspace.customPaneLayouts,
]

interface WorkspaceLayoutState {
  customPaneLayouts: readonly PaneLayoutDefinition[]
  sessions: readonly Session[]
}

interface SessionLayoutState {
  layoutId: PaneLayoutId
  placements: readonly PanePlacement[]
}

interface PanePlacement {
  paneId: string
  slotId: LayoutSlotId
}

interface Session {
  // existing fields...
  layout: SessionLayoutState
  panes: Pane[]
}

也就是说，用户自定义 layout definition 要跟 workspace/session snapshot 一起持久化；session 自己选择某个 layoutId，并记录 pane 到 slot 的 placement。这样 restore 时可以先把 prebuilt 和 customized 合成 registry， 再恢复每个 session 的当前 layout。

命名空间规则：prebuilt layout 使用固定 id，例如 single、grid3x2；用户自定义 layout 使用 custom:<uuid>。合并时不允许 custom 覆盖 prebuilt， 这样升级 app 时内置布局语义不会被用户数据 shadow 掉。

迁移策略：第一步可以继续让旧的 Session.layout: LayoutId 存在，并用 pane index 隐式映射到 addOrder。但目标模型必须把 pane identity 和 slot placement 拆开。

Registry / Management 层职责

VIM-151 已经把一部分 layout 规则从 UI 组件里抽到了 src/features/terminal/layout-registry。所以这里不是从零开始重构； 下一步是在现有 registry 基础上，把它从“prebuilt layout registry” 推进到“prebuilt + user customized layout management”。

职责 VIM-151 状态 剩余工作
layout id / 文案 / pill 顺序 已完成：LAYOUT_IDS、LAYOUTS、 LAYOUT_CYCLE 已集中到 layout registry。 需要让 registry 接受 runtime catalog： [...PREBUILT_PANE_LAYOUTS, ...workspace.customPaneLayouts]。
prebuilt layout specs 已完成：single / vsplit / hsplit / threeRight / quad / grid3x2 已作为 prebuilt specs 管理。 需要把 specs 升级成通用 PaneLayoutDefinition， 用 tracks + slot rect 表达，而不是只服务固定 enum。
ratio model 已完成：ratio 已从单个 row/col 值扩成 { cols: number[], rows: number[] }，能支撑多 track。 需要把用户可理解的 units 模型接入 runtime， 例如 24-unit 转成 ratio/fr。
displayed layout menu 已完成：top chrome 里已有 displayed-layout floating menu， 可以控制哪些 prebuilt layout pill 显示。 需要扩展为 layout management 入口：显示 prebuilt + customized， 并提供创建、编辑、删除 custom layout 的 plane。
autoshrink 已完成：当前 pane removal 的 shrink 规则已集中到 autoShrinkLayoutFor。 需要区分 prebuilt policy 和 custom policy。custom layout 里 empty slot 应该是合法状态，不能一关 pane 就强制坍缩 layout。

restore / persisted shape 部分完成：restore 已能识别当前 prebuilt layout id，并恢复 grid3x2。 需要持久化 customPaneLayouts 和 layoutId + placements，否则 custom layout reload 后无法解析。
Canonical API

PaneLayoutRegistry

它可以是 class，也可以是纯函数模块。关键是调用方只能问 registry： “这个 layout 是什么、是否有效、应该怎么渲染、pane 能不能加、关 pane 后怎么处理”。 

interface PaneLayoutRegistry {
  getDefinition(id: PaneLayoutId): PaneLayoutDefinition | null
  validateDraft(draft: DraftPaneLayoutDefinition): ValidationResult
  resolveGrid(def: PaneLayoutDefinition): ResolvedGrid
  resolveDividers(def: PaneLayoutDefinition): readonly DividerSpec[]
  resolveRatios(def: PaneLayoutDefinition): LayoutRatios
  resolveIcon(def: PaneLayoutDefinition): LayoutIconModel
  chooseLayoutAfterPaneRemoval(
    state: SessionLayoutState,
    nextPaneCount: number
  ): SessionLayoutState
}
Registry owns policy

不要再散落 switch

  • layout id 合法性判断
  • builtin 与 workspace custom 的合并顺序
  • displayed layout menu 可见项
  • layout icon / glyph 生成
  • autoshrink 或 empty-slot 策略
  • divider 边界和 ratio arrays
  • restore fallback 到 single 的规则

Validator：什么是有效 layout

规则 建议 v1 行为 原因
行列数量 1..4 tracks，最多 16 slots 覆盖 2x3、3x2、3x3、4x4，同时避免 first PR 复杂度失控
track units 必须是有限正数 ratio 和 CSS fr 不能接受 0、负数、NaN；UI 用 24 份单位表达比例
slot rect 整数、在 bounds 内、span 大于 0 保证可稳定映射到 grid area
重叠 禁止重叠 否则 pane focus、divider hit-test、placement 都不确定
洞 v1 禁止洞，要求完整覆盖网格 后续可以明确设计 spacer slot；现在不要让 empty area 语义混乱
addOrder 必须包含每个 slot 一次且只一次 新 pane 添加位置必须可预测
slot id 全局唯一且 stable pane placement 需要在 rename / resize / restart 后保持稳定

Customization Plane 应该怎么接入

UI state

只维护 draft

配置面板可以有 selection、hover、drag ghost、invalid preview、undo stack。 这些都属于 DraftPaneLayoutDefinition，不应该直接写进 session。 

interface DraftPaneLayoutDefinition {
  base?: PaneLayoutId
  title: string
  columns: DraftTrack[]
  rows: DraftTrack[]
  slots: DraftSlot[]
  selectedSlotId?: LayoutSlotId
}
Commit path

Apply 时才进入 registry

  • 用户在 plane 里编辑 draft
  • validator 给出 inline errors 和 preview warnings
  • Apply 后生成 canonical definition
  • 写入当前 workspace/session snapshot 的 customPaneLayouts
  • 运行时用 [...prebuilt, ...customized] 重新组装 layout list
  • 当前 session 切换到新的 layoutId 并写入 placement

持久化建议

prebuilt layout 定义继续在代码里；用户自定义 layout 定义随 workspace/session snapshot 一起持久化。restore 时先组装 [...PREBUILT_PANE_LAYOUTS, ...persisted.customPaneLayouts]， 再用每个 session 的 layoutId 和 placements 恢复 pane 布局。 

interface PersistedWorkspaceLayoutSnapshot {
  schemaVersion: 1
  customPaneLayouts: readonly PaneLayoutDefinition[]
  sessions: readonly PersistedWorkspaceSessionShape[]
}

interface PersistedWorkspaceSessionShape {
  id: string
  projectId: string
  layoutId: PaneLayoutId
  placements: readonly PanePlacement[]
  workingDirectory: string
  active: boolean
  panes: PersistedWorkspacePaneShape[]
}

fallback：restore 时如果 layoutId 找不到、definition 无效、 或 placements 指向不存在的 slot，则降级到 single 或最近的 builtin grid，并记录 warning。不要让坏 layout 阻塞 session restore。

实现顺序

阶段 目标 为什么先做
PR 1 引入 PaneLayoutDefinition、validator、derived grid/divider helpers 先把 runtime SSoT 定住，避免 UI 设计绑死错误结构
PR 2 把现有 builtin layouts 迁移到 definition model 证明新模型能无回归表达 single/vsplit/hsplit/quad/grid3x2
PR 3 workspace/session snapshot 增加 customPaneLayouts 与 placement restore 兼容层 自定义布局必须能 reload，不然只是临时 UI
PR 4 增加 Workspace Customization Plane 此时 UI 只是在编辑 draft，不需要发明自己的 runtime model
PR 5 增加 templates：2x3、3x3、4x4、Main + 3 stack 模板只是 definition factory，不再污染 SplitView

需要特别防范的耦合

风险

pane count 驱动 layout

旧 autoshrink 是按 pane 数决定 layout。自定义布局里 empty slot 是合法状态， 所以不能一关 pane 就自动坍缩 layout。autoshrink 应该变成 registry policy， 并且默认只作用于 builtin presets。

风险

layout id 到处 switch

新 layout 不应该要求修改 SplitView、SplitDividers、 LayoutGlyph、restore、shortcut 多个模块。调用方应只消费 registry 的 resolved model。

风险

customization draft 泄漏到 runtime

draft 可以暂时 invalid；runtime 不行。Apply 之前必须经过 validator， 且生成 canonical definition。

风险

只持久化 layoutId

对 custom layout 来说，只有 id 不够。必须把 customPaneLayouts 跟 workspace/session snapshot 一起持久化， 否则 reload 后 id 没有 definition 可解析。

建议结论：先做 SSoT 数据结构和 validator，再做 customization plane。UI 可以大胆， 但 runtime 必须保守：只有合法、可派生、可恢复的 layout definition 能进入 session。
