Technical Note

Adding a 3x2 pane layout pill

This note maps how terminal-zone pane layouts work today, where the current implementation is tightly coupled to a 5-layout / max-4-pane world, and what must change before adding a true 6-pane 3x2 layout to the layout pill group.

Scope: terminal zone + top chrome Current date: 2026-06-17 Recommended id: grid3x2 Capacity target: 6 panes
Current Layout Set 5 layouts single, vsplit, hsplit, threeRight, quad
Hidden Constraint 1 column ratio plus 1 row ratio per layout in the resize model
Main Risk Render path is extendable but divider and ratio plumbing are not 3-column ready

Current structure

UI entry point

Top chrome pill group

WorkspaceView.tsx renders LayoutSwitcher in the top chrome. The switcher maps directly over Object.values(LAYOUTS), so adding a new layout record automatically adds a new pill.

  • src/features/workspace/WorkspaceView.tsx
  • src/features/terminal/components/LayoutSwitcher/LayoutSwitcher.tsx
Session model

Layout is session state

Each session carries a layout: LayoutId. The active session layout is mutated through setSessionLayout(sessionId, layoutId) in useSessionManager.

  • src/features/sessions/types/index.ts
  • src/features/sessions/hooks/useSessionManager.ts
Render path

TerminalZone -> SplitView

TerminalZone renders one SplitView per session, hides inactive sessions, and forwards pane focus/add/remove handlers. SplitView chooses visible panes, places them into CSS grid slots, and mounts divider handles only for the active session.

  • src/features/workspace/components/TerminalZone.tsx
  • src/features/terminal/components/SplitView/SplitView.tsx
Layout source of truth

LAYOUTS record

LAYOUTS currently defines name, capacity, base grid template, and area matrix. It also implicitly defines the keyboard cycle order because usePaneShortcuts derives LAYOUT_CYCLE from record insertion order.

  • src/features/terminal/components/SplitView/layouts.ts
  • src/features/terminal/hooks/usePaneShortcuts.ts

Important: adding a record to LAYOUTS is enough to render a new pill, but it is not enough to safely support a real 3x2 layout end-to-end.

How the layout engine is arranged today

WorkspaceView
  -> LayoutSwitcher
     -> Object.values(LAYOUTS)
        -> onPick(layoutId)
           -> useSessionManager.setSessionLayout(...)

WorkspaceView
  -> TerminalZone
     -> SplitView(session.layout)
        -> LAYOUTS[layoutId]
        -> selectVisiblePanes(session.panes, capacity)
        -> resolveGrid(layoutId, currentRatios)
        -> render pane slots by index as p0..pN
        -> render SplitDividers(layoutId, ratios)
Behavioral details
  • Slots are placed by visible pane index, not by pane.id. The wrapper style is gridArea: p${i}.
  • If a session has more panes than the active layout capacity, selectVisiblePanes keeps the active pane visible by replacing the last visible slot.
  • Empty slots only render when onAddPane is present.
  • Divider handles only mount for the active session and only after the container has measurable size.

Coupling problems to watch

Area Current assumption Why it matters for 3x2
LayoutId Union is fixed to 5 values. New layout touches the session type, restore whitelist, test snapshots, and every switch over layout ids.
LayoutShape.capacity 1 | 2 | 3 | 4 A 3x2 layout needs 6, so this type must widen immediately.
LayoutRatios Exactly one col and one row. This is the biggest structural blocker. A 3-column layout needs two independent vertical divider positions, not one.
resolveGrid() Builds templates from --split-col, --split-col-end, --split-row, --split-row-end. That variable scheme only expresses two pane tracks per axis. A true 3x2 needs a different track model.
SplitDividers Covers four hard-coded layouts and reuses one column binding for segmented bars in quad. 3x2 needs two logical column bindings plus one row binding, likely rendered as 5 resize handles total.
useSplitDivider() Accepts only '--split-col' or '--split-row'. It cannot currently drive --split-col-0, --split-col-1, etc.
Keyboard cycle Cmd/Ctrl+\ cycle comes from LAYOUTS insertion order. Inserting 3x2 changes the shortcut sequence for every user. That should be intentional, documented, and tested.
Auto-shrink autoShrinkLayoutFor() only knows 1, 2, 3, and 4-plus pane cases. Closing from 6 to 5 panes has no current policy. Without a rule, the layout downgrade path becomes arbitrary.
Restore / hydration KNOWN_LAYOUT_IDS in restore code only allows the current 5 ids. A persisted 3x2 session would currently fall back to single.
Tests Several tests assert "5 buttons", "5 canonical layouts", or only current layout switch cases. Adding the pill without updating the test matrix will create a broad but expected test break.

Recommended shape for the new layout

Suggested identifier

grid3x2

I would avoid naming it by pane count alone. grid3x2 is explicit about geometry, reads cleanly beside threeRight, and leaves room for future grid2x3 or asymmetric 6-pane layouts if needed. 

type LayoutId =
  | 'single'
  | 'vsplit'
  | 'hsplit'
  | 'threeRight'
  | 'quad'
  | 'grid3x2'
Proposed slot map 
areas: [
  ['p0', 'vdiv0', 'p1', 'vdiv1', 'p2'],
  ['hdiv', 'hdiv', 'hdiv', 'hdiv', 'hdiv'],
  ['p3', 'vdiv2', 'p4', 'vdiv3', 'p5'],
]

This keeps the same "segmented vertical handles around a full-width horizontal divider" approach used by quad, but doubles the logical column divider count.

Implementation implications

Safe, local changes
  • Add grid3x2 to LayoutId.
  • Add a new LAYOUTS.grid3x2 entry and include its display name.
  • Extend LayoutGlyph to render two vertical separators plus one horizontal separator.
  • Extend restore whitelist logic in groupSessionsFromInfos.ts.
Non-local changes
  • Replace the current { col, row } ratio model with a layout-aware shape that supports multiple vertical splits.
  • Generalize useSplitDivider() so a divider can target arbitrary CSS variable names.
  • Add a dedicated grid3x2 branch to resolveGrid() and SplitDividers.
  • Decide the shrink path for 6 -> 5 -> 4 panes. My recommendation: 6/5 stay grid3x2, 4 falls to quad, 3 to threeRight, 2 to vsplit or preserve hsplit if that was the current layout.

Recommendation: do not treat this as a pure icon + enum change. If the product intent is a real interactive 3x2 layout, the resize model should be refactored first or in the same PR.

Bottom line

The terminal-zone layout system is well organized around SplitView, but its resize architecture is still tuned for layouts that need at most one horizontal split and one logical vertical split. A new 3x2 pill is straightforward; a production-ready 3x2 layout is a small structural extension, not just a new case statement.

If the next step is implementation, the cleanest sequence is: 1) widen the ratio/divider model, 2) add grid3x2 render + glyph support, 3) update shrink/restore policies, 4) repair the test matrix.
