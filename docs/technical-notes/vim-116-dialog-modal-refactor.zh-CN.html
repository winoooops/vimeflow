2. Dialog 和 ModalDialog 应该如何分工

Dialog 是语义和结构

Web / ARIA 层面，dialog 只说明“这是一个独立交互区域”。它可以是 anchored popover，也可以是 inline composer，也可以是全屏 modal。 因此 Popover 继续保留：它是 anchored dialog card， 不是 app-level modal。

新 primitive 如果命名为 Dialog，语义上仍然合理， 但文档必须明确：它负责的是 app-level modal shell，不替代 Popover。

ModalDialog 是行为集合

modal 的关键不是视觉上像弹窗，而是背景不可交互、焦点被限制在内部、 Escape / backdrop 有明确 dismiss 策略、关闭后能恢复原焦点。 这些行为现在散落在 feature 里，应该集中到全局组件。

代码命名建议先用 Dialog，因为项目里的 public primitive 都是短名；如果未来确实需要 non-modal app dialog，再扩展 modal={false} 或拆出 ModalDialog。

结论： 不要把 Popover 改造成全屏 modal。它依赖 anchor 定位和 base/floating，适合小型 card。新的 Dialog 应该拥有自己的 overlay substrate，专门处理 viewport-level modal。

3. 和 floating surface 的共享边界

你的疑问是关键：非阻塞 dialog 和 menu / popover 的确共享很多逻辑， 但共享点在“anchored floating surface”，不是在“所有 overlay 都做成一个 Dialog”。因此规划里要把 substrate 分清。

层 负责什么 使用者 规划决策
base/floating anchor / cursor 定位、flip / shift、outside press、portal、 list navigation、anchored focus manager。 Dropdown、Menu、Popover 保持为非阻塞 anchored surface 的默认路径。缺功能时优先扩展 Popover 或 Menu。
Dialog viewport-level fixed overlay、scrim、modal stacking、focus trap/restore、Escape/backdrop dismiss、aria-modal。 UnsavedChangesDialog、CommandPalette， 后续 settings-style app dialog。 新增 public primitive，但不吞并 Popover。它只处理 全屏 app modal shell。
共享视觉语言 Lens glass panel、backdrop 色彩、z-index 策略、motion timing、 focus/dismiss 纪律。 所有 overlay primitive 可以抽 token / className helper，但不要让 Dialog 依赖不需要的 anchor-positioning machinery。
规划约束： 如果一个 surface 需要 anchor 或 cursor 定位，它不应该使用 Dialog；如果一个 surface 需要挡住整个 workspace 并限制背景交互， 它不应该使用 Popover。

4. 共同逻辑应该放进全局组件

Stacking 与遮罩

  • 统一 fixed inset-0 root 和 z-[100] 层级。
  • 统一 backdrop 颜色、blur、点击 dismiss 策略。
  • 支持 center / top 两种 placement，覆盖 Unsaved 和 Palette。

Focus 与键盘

  • 打开时记录 previous focus，关闭后恢复。
  • 支持 initialFocusRef，没有则聚焦第一个 tabbable 元素。
  • 统一 Tab trap、Shift+Tab 循环、Escape dismiss。

ARIA 与可访问性

  • 统一 role="dialog" 和 aria-modal="true"。
  • 支持 aria-label 或 aria-labelledby。
  • 支持 aria-describedby，让确认弹窗能被完整读出。

Motion 与面板壳层

  • 集中 AnimatePresence、backdrop fade、panel spring。
  • 统一 glass panel chrome，避免每个 feature 复制 className。
  • 面板尺寸走受控的 size / placement，不开放任意皮肤。

Dismiss 策略

  • closeOnBackdrop 和 closeOnEscape 默认开启。
  • dismissDisabled 支持保存中、提交中、危险操作中锁定关闭。
  • 所有关闭都通过同一个 onOpenChange(false) 或 onDismiss。

测试边界

  • primitive 测 focus trap、Escape、backdrop、aria 和 restore focus。
  • feature 测业务文案、按钮回调、query/results 或 terminal lifecycle。
  • 避免每个 consumer 重测 primitive 已覆盖的键盘细节。

5. 不应该放进全局组件的东西

不要放入 Dialog 原因 保留在哪里
“保存 / 丢弃 / 取消”的业务动作 这是 editor 的文件状态语义，不是 dialog 的通用能力。 UnsavedChangesDialog consumer
Command palette 的 query、selectedIndex、activeDescendant 这是 command registry 和 combobox/listbox 行为。 CommandPalette 及其 hook
xterm attach、focusTerminal、Tab pass-through 这是 terminal workspace overlay 的生命周期和输入模型。 BurnerTerminalPopup
任意 panel className 皮肤 会重新打开 VIM-116 要关闭的视觉漂移。 Dialog 只允许尺寸、placement、布局级 className 逃生口
anchor positioning / cursor positioning 这些已经由 base/floating 和 Popover / Menu.Context 解决。 Popover、Menu、Dropdown

6. 建议的 public API 草案

这个 API 的目标是：足够 generic，可以承载 Unsaved、CommandPalette 和后续 Settings dialog；同时不把 feature 逻辑塞进 primitive。 

interface DialogProps {
  open: boolean
  onOpenChange: (open: boolean) => void
  placement?: 'center' | 'top'
  size?: 'sm' | 'md' | 'lg' | 'xl'
  closeOnBackdrop?: boolean
  closeOnEscape?: boolean
  dismissDisabled?: boolean
  restoreFocus?: boolean
  initialFocusRef?: React.RefObject<HTMLElement | null>
  'aria-label'?: string
  'aria-labelledby'?: string
  'aria-describedby'?: string
  testId?: string
  backdropTestId?: string
  children: React.ReactNode
}

// 可选结构 helpers，保持 spacing / divider / footer 对齐。

API 原则： Dialog owns shell and behavior；consumer owns content and callbacks。只把跨 feature 重复、可测试、和 accessibility 相关的逻辑放进 primitive。

7. 与 src/components 架构保持一致

现有模式

Dropdown、Menu、Popover 是 public primitive，底层共享 src/components/base/floating。Button、 IconButton、ToolbarButton 共享 base/button。feature 只能 import public primitive， 不能 import base/**。

Dialog 应该遵守

Dialog.tsx 平铺在 src/components/， 和其他 public primitive 同级。若需要 low-level focus/overlay helper， 放在 src/components/base/dialog/，继续遵守 base/** package-private 边界。

不建议让 Dialog 直接复用 SurfacePanel： 现在 SurfacePanel 需要 FloatingContext， 强行复用会让 viewport modal 依赖 anchor 定位栈。更合理的是抽小型 chrome helper，或在 Dialog 中保持同一套 token。

这次 branch 也新增了 src/components/README.md 和 src/components/AGENTS.md，把这些边界直接放进组件目录， 以后做新的 shared primitive 时不需要重新从设计文档里拼规则。
