VIM-342 · Reviewed Architecture · 2026-07-18

Nonce-Scoped Reply and Review Recovery 基于 Nonce 的 Reply 与 Review 定向恢复

Recover one missed structured reply or delegated review when its pane becomes active again—without weakening transcript replay protection, starting inactive watchers, or adding persistence to the pane-switch fix. 当 pane 再次激活时，只恢复对应的结构化回复或 review；不削弱 transcript replay 防护，不启动 inactive watcher，也不把持久化重构塞进 pane-switch 修复。

01 · Core decision / 核心决策

Recover requested data; never replay the event stream.

只读取仍在等待的 nonce，不重新广播历史事件。

Keep the flood guard

Codex and Claude keep their current if !replay_done { return; } behavior. General transcript replay still suppresses historical agent-reply or agent-review output.

保留 replay 防洪机制

Codex 与 Claude 继续保留现有的 replay_done guard。普通 transcript replay 仍然不能发出历史 agent-reply 或 agent-review。

04 · Delivery semantics / 投递语义

The event is not “only once.” The effect is.

实时事件与恢复结果可以竞争，但 UI 结果只能生效一次。

Why duplicates are safe / 为什么重复投递安全

The live listener and targeted recovery may both observe the same reply. Both call the same synchronous handleReply. JavaScript runs them sequentially: the first deletes the matched handle; the second no longer finds a pending target.

live listener 与 recovery 可能同时拿到同一条回复，但 JavaScript 会顺序执行。第一个调用消费 handle，第二个调用找不到 pending target，因此不会重复挂载。

07 · Review checklist / 验收清单

What to verify before implementation approval

这些条件全部通过后，VIM-342 才算完成。

  • Pane A reply or review returns to pane A, never pane B.
    回复只回到原始 pane A。
  • Repeated switching creates no duplicate.
    反复切换不会重复挂载。
  • Historical nonces are not requested.
    未 pending 的历史 nonce 不会被查询。
  • Normal replay still emits no historical reply.
    普通 replay 仍保持防洪。
  • Recovery never starts a watcher.
    恢复流程不会启动 watcher。
  • Fast reply cannot beat pending registration.
    快速回复不能抢在 pending 登记之前。
