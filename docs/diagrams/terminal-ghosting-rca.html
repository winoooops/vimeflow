Vimeflow · Root-Cause Investigation

Anatomy of a Ghost

Three terminal glitches that looked like one bug were really four, living in four different layers of the stack. Here is how the theory was built — and why the fix that proved the cause was deliberately not the fix that shipped.

4 root causes 6 hypotheses tested 2 independent analyses (Claude + Codex) WebGL preserved

01One symptom name, four real bugs

Everything showed up as “the terminal renders wrong.” That framing is a trap: it invites a single fix for what are actually independent faults. The first job was to split the symptom into distinct, separately-triggered behaviours.

① Font ② Locale ③ GPU ④ rAF

The same pixels, four different points of failure — from the shell that emits the bytes down to the GPU process that paints them.

02The method: theory before patch

The investigation opened with a deliberately humbling rule, because the early days had been a guess-and-ship loop that fixed nothing.

The hard gate. Write no fix until two independent analyses — Claude’s and an independent codex exec reading the same evidence — agree on the mechanism. Every earlier “fix” (a Canvas2D swap, clearTextureAtlas(), backgroundThrottling, a premature focus-repaint) was demoted to an unverified guess and had to be confirmed or refuted with evidence, not asserted.

Triage by trigger, not by looks

The breakthrough wasn’t staring at screenshots — it was asking “what makes each artifact disappear?” Two glitches shared one trigger (a window switch) but had opposite survival characteristics under software rendering. That split one apparent bug into two causes.

A diagnostic is not a fix

Disabling hardware acceleration made the corruption vanish — a perfect proof of where the fault lived. Shipping it would have thrown away GPU rendering app-wide and reversed a documented decision. The proof and the cure are different objects.

03Splitting the ghost in two

“Ghosting” after switching windows mid-stream was the hardest report. It turned out to be two causes wearing one costume. The decisive experiment was running the exact same session under software rendering.

One experiment, a clean fork. What dies without a GPU was a GPU bug; what lives without a GPU was never about the GPU at all.

A second tell corroborated cause ④: selecting text with the mouse made the duplicate rows snap clean. A selection forces a repaint — so the buffer was always correct; only the paint was stale. That pointed straight at xterm’s render scheduler.

04The four causes, end to end

For each: the symptom, the layer it lived in, the path from observation to theory, the evidence that nailed it, and the smallest reversible fix that followed.

① Font · renderer CSS

Prompt glyphs misaligned, text shoved right

Layer: renderer font stack · terminalFont.ts, index.css

How the theory formed

Measured the actual glyph advances. The installed Hack Nerd Font is a non-Mono variant — its Powerline glyphs ink wider than one cell — and the font stack leaned on whatever Nerd Font happened to be installed, so the source varied machine to machine. Cross-referenced how kitty (symbol_map), Ghostty and Windows Terminal solve it: map a codepoint range to one dedicated face.

Evidence

Glyphs only misaligned for the Powerline / Nerd codepoint range; ordinary text was fine. The advance differed across machines with different fonts installed — a sourcing problem, not a layout one.

Fix → bundle a Vimeflow Nerd Symbols face and scope it with CSS unicode-range (the web equivalent of symbol_map); reorder the stack to put it first. 76d1524
② Locale · Rust PTY spawn

Cursor flung to the far right while editing

Layer: PTY spawn environment · commands.rs

How the theory formed

Noticed it only happened in GUI launches (dock / Finder / electron:dev), never in Terminal.app. Those GUI launches inherit no LANG / LC_*, so the shell runs in the C locale and byte-counts multibyte glyphs. zsh then measures each Powerline / Nerd glyph as ~3 cells, mis-computes the line width, and desyncs the cursor.

Evidence

wcwidth returns the wrong width for these codepoints only in the C locale; Terminal.app and iTerm2 set a UTF-8 locale and never exhibit it. The fault is environmental, upstream of any rendering.

Fix → inject LC_CTYPE (only the character-width knob, leaving message/collation language alone) when no UTF-8 locale is inherited — en_US.UTF-8 on macOS, C.UTF-8 elsewhere. 717cc9a
③ GPU · Chromium GPU process

Garbled “alien” glyphs after a window switch

Layer: Chromium GPU process · electron/main.ts

How the theory formed

Ran the hypothesis gauntlet. A Canvas2D swap didn’t help (same GPU process). Clearing the texture atlas didn’t help. The addon/version pairing was correct. Then disabling hardware acceleration made the corruption vanish entirely — isolating the fault to the GPU layer: Chromium reclaims an occluded window’s GPU resources and scrambles xterm’s cached glyph textures.

Evidence

Corruption present under WebGL and Canvas (both paint through the GPU process); gone under software rendering. The trigger is occlusion — covering the window — which is exactly when Chromium backgrounds it.

Fix → keep WebGL + hardware acceleration, but stop the reclaim with disable-backgrounding-occluded-windows + disable-renderer-backgrounding. f99742c
④ rAF · xterm render scheduler

Duplicate / overlapping rows + stray “?”

Layer: xterm render loop · Body.tsx

How the theory formed

Read @xterm/xterm@6.0.0 directly: rendering runs on a debounced requestAnimationFrame loop (RenderDebouncer → rAF → refreshRows), and xterm listens only for pagehide — never visibilitychange. Chromium throttles rAF for an occluded window, so rows that streamed in while you were away never flush, and nothing repaints them on return.

Evidence

Renderer-agnostic and survives software rendering (so: not the GPU). A text selection clears it instantly — proof the buffer is right and only the paint is stale.

Fix → force a full refresh(0, rows-1) on window focus and on document visibilitychange → visible — flush the stale rows on the exact event xterm ignores. f8a8577

05The hypothesis ledger

Every candidate was marked confirmed, refuted, or diagnostic-only with evidence — not opinion. Claude and an independent codex exec reached the same conclusions before any fix was written.

Hypothesis Verdict Basis
GPU texture reclaim on occlusion Confirmed Vanishes under software rendering; renderer-agnostic.
rAF render-throttle, no visibility repaint Confirmed xterm source: rAF loop + no visibilitychange; survives software render; selection clears it.
C-locale glyph-width miscount Confirmed GUI launch inherits no UTF-8 locale; wcwidth wrong in C locale.
Nerd Font glyph sourcing Confirmed Misalign scoped to symbol codepoints; advance varies by machine.
Canvas2D-vs-WebGL renderer swap Refuted Canvas paints through the same GPU process — corrupts identically.
Texture-atlas corruption (clearTextureAtlas) Refuted Residual survives software rendering, where there is no atlas to clear.
Renderer addon / version mismatch Refuted webgl 0.19.0 is the stable pair for xterm 6.0.0.
PTY → xterm data-path double-write Refuted Drain is mount-time only; “selection fixes it” ⇒ buffer correct.
Disable hardware acceleration Diagnostic Proves the GPU layer — but reverses the documented WebGL decision, so rejected as a fix.

06Intentionally landing the fix

With the mechanism agreed, the rule was: the smallest reversible change that follows directly from each cause — and not one that contradicts the architecture.

WebGL stays primary. The roadmap documents WebGL as the renderer of choice for large-agent-output throughput. The diagnostic that proved cause ③ (disabling hardware acceleration) was therefore explicitly not shipped; the two occlusion switches keep WebGL and remove only the reclaim that corrupts the cache.

Four atomic commits, not one mega-fix. Each cause is independently testable and revertable — a direct product of isolating each mechanism before touching code.

One honest deferral. The backend decodes each PTY read with String::from_utf8_lossy, which can split a multibyte glyph across read boundaries and emit a stray ? (U+FFFD). Codex flagged it as a third contributor to the stray-? artifact. It is logged as a follow-up — a carry-over decoder — rather than rushed.

What only a human can close. The fixes verify green across lint, type-check, and the full unit + Rust suites. But the final before/after under real OS-window occlusion can’t be staged by automation — CDP can’t occlude one native window with another, and the shipped switches suppress the very throttle that produces the artifact. That last observation is, by its nature, a person at the machine switching windows while output streams.
